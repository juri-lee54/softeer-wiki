## 식별자와 키

### 식별자

- 엔티티는 인스턴스들의 **집합**
- 집합 안에서 각 인스턴스를 **유일하게 구분**하는 속성(들)이 필요
- 이 "구분자"가 바로 **식별자(Identifier)** = DB에서는 **키(Key)**

<br>

**회원 엔티티**

| **회원번호** | **이름** | **이메일** |  |
| --- | --- | --- | --- |
| 1001 | 홍길동 | a@x.com | ← 1001 로 이 한 줄을 콕 집을 수 있음 |
| 1002 | 김철수 | b@x.com |  |

<br>

**식별자의 조건 2가지**

- 식별자가 되려면 두 가지를 만족해야 한다.

| **조건** | **의미** | **예** |
| --- | --- | --- |
| **유일성 (Unique)** | 값이 겹치지 않는다 | 회원번호 1001은 단 한 명 |
| **최소성 (Minimal)** | 더 줄이면 유일성이 깨진다 | 꼭 필요한 속성만으로 구성 |
- 유일성만 있고 최소성이 없으면 → 군더더기가 붙은 키
- 둘 다 있어야 → 깔끔한 "후보키"

<br>

### 키의 포함관계

!image.png

**슈퍼키 ⊇ 후보키 ⊇ (기본키 ∪ 대체키)**

<br>

### 슈퍼키 (Super Key)

- **유일성은 만족**하지만 **최소성은 신경 안 쓴** 키
- 즉, 유일하게 구분만 되면 OK (군더더기 속성이 붙어도 됨)

```
학생 테이블에서…
  { 학번 }                  → 유일 O  (군더더기 없음)
  { 학번, 이름 }            → 유일 O  (이름은 군더더기)
  { 학번, 이름, 학과 }      → 유일 O  (다 군더더기)
```

- 위 세 개 모두 **슈퍼키**
- 이 중에서 군더더기를 빼면 -> 후보키

<br>

### 후보키 (Candidate Key)

- **유일성 + 최소성 모두** 만족하는 키
- "기본키가 될 자격이 있는 후보들"

```
학생 테이블
  { 학번 }       → 유일 O, 최소 O  → 후보키 O
  { 이메일 }      → 유일 O, 최소 O   → 후보키 O
  { 학번, 이름 }   → 최소 X          → 후보키 X (슈퍼키일 뿐)
```

- 한 테이블에 후보키가 **여러 개** 있을 수 있다
- 이 중 하나를 대표로 뽑으면 → 기본키

<br>

### 기본키 & 대체키

- **기본키(Primary Key)**: 후보키 중 **대표로 선정한 하나**
- **대체키(Alternate Key)**: 후보키인데 기본키로 **안 뽑힌 나머지**

```
후보키 = { 학번 }, { 이메일 }

   학번  →  기본키(PK)로 선정 ✅
   이메일 →  대체키 (여전히 유일하지만 대표는 아님)
```

- 공식: **후보키 = 기본키 + 대체키**
- 대체키는 보통 DB에서 **`UNIQUE`** 제약으로 살려둔다
    - 대체키를 언제 만들고 만들지 않을지 선택해야 함
        - 장점
            - 데이터 정합성 GOOD
        - 단점
            - UNIQUE 인덱스 자체가 저장 공간 낭비
            - INSERT 할 때마다 검사 → INSERT 성능 떨어짐
        - 데이터 정합성 - 저장 공간과 성능의 트레이드 오프

<br>

### 외래키 (Foreign Key)

- 다른 엔티티의 **기본키를 참조**하는 속성
- 관계(1:N 등)를 테이블에서 표현하는 수단

!w:430

- FK 덕분에 "이 주문이 누구 것인지" 추적 가능
- 존재하지 않는 회원번호는 못 넣게 막아줌(**참조 무결성**)
- "참조하는 데이터는 반드시 존재해야 한다"
- 부모 테이블의 키값 or **NULL** 가능
- 실무에서는 성능을 위해 FK 제약 조건을 안 넣는 경우도 많음
- FK 제약 조건의 2가지 물리적 의미
    - 2개의 테이블을 검사함
    - FK 제약조건을 건 컬럼에 인덱스를 설정함

<br>

### 키 종류 정리

| **키** | **유일성** | **최소성** | **한 줄 정의** |
| --- | --- | --- | --- |
| **슈퍼키** | O | △ | 유일하게 구분되면 OK |
| **후보키** | O | O | 슈퍼키 중 군더더기 없는 것 |
| **기본키** | O | O | 후보키 중 대표 하나 |
| **대체키** | O | O | 후보키 중 PK 빼고 나머지 |
| **외래키** | - | - | 다른 테이블 PK를 참조 |

면접 단골: "슈퍼키와 후보키의 차이는?" → **최소성**

<br>

**보조식별자(Secondary)**

- 기본키 말고 **인스턴스를 식별할 수 있는 또 다른 속성**
- 주로 **다른 엔티티와 연결고리** 역할이 필요할 때 정의
- 실무에서는 **`UNIQUE INDEX`** 로 구현되는 경우가 많다
    - 대체키 중 하나에서 UNIQUE 인덱스 설정

```
-- 회원번호가 PK지만, 이메일로도 찾고 싶다
CREATE UNIQUE INDEX ux_member_email ON member(email);
```

- 주의: 보조식별자가 **항상 UNIQUE인 것은 아니다** (보조식별자 ≠ UNIQUE INDEX)

<br>

### 자연키 vs 대리키 (중요)

키의 "출처"로 나누면 두 종류가 있다.

| **구분** | **본질식별자 (자연키)** | **인조식별자 (대리키)** |
| --- | --- | --- |
| **영어** | Natural Key | Surrogate Key |
| **정체** | 업무상 의미가 있는 값 | 시스템이 인위적으로 만든 값 |
| **예시** | 주민번호, 이메일, 학번 | `auto_increment` ID, UUID |
| **의미** | 데이터 자체에 뜻이 있음 | 그냥 일련번호, 뜻 없음 |
- 자연키에서 대리키로 넘어가는 추세
    - 주민번호 대신 auto_increment, UUID 사용

<br>

### 본질식별자 (자연키)

- **내용(의미)상의 진정한 식별자**
- 집합의 본질을 설명하는 "의미상의 주어" — **누가/무엇을/언제**
- 특별한 하자가 없으면 **이게 곧 기본키**가 된다

```
보험계약 엔티티의 본질식별자
  누가(WHO)    → 고객번호
  무엇을(WHAT) → 보험상품코드
  언제(WHEN)   → 계약일자
  → 이 셋의 조합이 한 건의 계약을 구분
```

- 장점: 의미가 있어 직관적
- 문제: 자주 깨진다

<br>

**자연키가 깨지는 순간들**

본질식별자가 "논리적으로는" 맞아도 현실에서 문제가 생긴다.

1. **유일성 실패**: 동일 고객·동일 상품·동일 일자에 동시 가입?
    
    → 셋을 합쳐도 유일하지 않다
    
2. **너무 김 / 복잡함**: 키가 3~4개 컬럼이라 자식 테이블마다 다 끌고 다님
    
    → JOIN 조건이 매번 길어진다
    
3. **값이 바뀜**: 이메일·전화번호·학번처럼 나중에 변경될 수 있음
    
    → PK가 바뀌면 참조하던 FK가 전부 흔들림
    

**⇒** 이럴 때 등장하는 것이 **인조식별자(대리키)**

<br> 

### 인조식별자 (대리키, Surrogate Key)

- 본질식별자를 대체하려고 **인위적으로 만든 식별자**
- 의미 없는 값 — 그냥 유일하기만 하면 된다

```
CREATE TABLE contract (
  contract_id BIGINT AUTO_INCREMENT PRIMARY KEY,  -- 인조키
  customer_no VARCHAR(20),
  product_code VARCHAR(20),
  contract_date DATE,
  UNIQUE (customer_no, product_code, contract_date) -- 자연키는 UNIQUE로 보존
);넓게 보기코드 복사
```

- 흔한 형태: **`BIGINT AUTO_INCREMENT`**, **`UUID`**
    - UUID
        - 장점
            - ID를 순서대로 쓰면 간접적으로 유출되는 정보가 있어서 보안용으로 사용함
            - 분산 시스템에서 UNIQUE를 보장
        - 단점
            - 16byte - 데이터가 크다는 것이 문제 → 저장 공간 많이 차지
            - 정렬이 안 되어있음 - 시간 순서가 없음
                - 버전 4, 7 많이 사용
                - 버전 7는 시간순으로 개선됨
- 자연키는 버리지 말고 **`UNIQUE` 제약**으로 살려두는 게 좋다.

<br>

#### 인조키를 쓰는 이유 4가지

| **#** | **원칙** | **무슨 상황?** |
| --- | --- | --- |
| 1 | 유일성 보장 | 자연키만으로 유일성이 안 될 때 |
| 2 | 편의성·단순성 | 자연키가 길어 JOIN이 복잡할 때 |
| 3 | 변경 대비 | 자연키 값이 바뀔 수 있을 때 (이메일 등) |
| 4 | 자손이 많을 때 | 자식 테이블이 많아 키를 계속 상속시킬 때 |

"왜 인조키?"를 물으면 위 4가지 중 하나로 답할 수 있어야 한다

<br>

#### 인조키 판단 흐름 - "왜?"로 생각하기

!w:1100

- 모든 테이블에 **`id`** 박는 것도 정답은 아니다
- 다만 **실무 BE 기본값은 인조키(PK)** — 변경에 가장 안전하기 때문
- 자연키가 좋을 수도 있음

<br>

#### 인조키 주의 사항

- **불러줄 대상이 없으면** 인조키 PK는 불필요한 인덱스만 늘린다
    - 아무도 그 **`id`**로 조회하지 않는데 만들어두는 경우
- 자연키 자체가 짧고 안정적이면(예: 국가코드 **`KR`**, **`US`**) 그대로 PK가 나을 수도
- 인조키를 쓰더라도 **자연키의 유일성은 반드시 `UNIQUE`로 지켜라**
    - 안 그러면 같은 회원이 두 번 가입하는 중복이 슬그머니 들어온다

인조키는 "편하려고" 쓰는 것이지, 데이터 정합성을 포기하려고 쓰는 게 아니다

<br>

### 기본키(PK) 선정 기준

후보키가 여러 개일 때 무엇을 PK로 뽑을까?

| **기준** | **설명** |
| --- | --- |
| **NOT NULL** | PK는 절대 NULL 불가 |
| **불변성** | 한 번 정해지면 잘 안 바뀌는 것 |
| **단순성** | 짧고 단순할수록 좋음 (JOIN·인덱스 유리) |
| **대표성** | 이 엔티티를 가장 잘 대표하는 것 |
- 이메일 vs 회원번호 → 변경 가능한 이메일보다 **회원번호(또는 인조 id)** 가 PK
- 이메일은 대체키로 **`UNIQUE`** 처리

<br>

### DDL로 마무리 보기

```
CREATE TABLE member (
  member_no  BIGINT AUTO_INCREMENT,   -- 인조 기본키
  email      VARCHAR(255) NOT NULL,   -- 자연키 → 대체키
  name       VARCHAR(50)  NOT NULL,
  PRIMARY KEY (member_no),            -- 기본키
  UNIQUE KEY ux_email (email)         -- 대체키(보조식별자)
);

CREATE TABLE orders (
  order_no   BIGINT AUTO_INCREMENT PRIMARY KEY,
  member_no  BIGINT NOT NULL,
  FOREIGN KEY (member_no) REFERENCES member(member_no)  -- 외래키
);
```

- 외래키 제약조건은 이론적으로 쓰는 게 기본이나 실무에서 안 사용하는 추세

<br>

### 정리

- **식별자 = 키**: 인스턴스를 유일하게 구분 (유일성 + 최소성)
- **슈퍼키 ⊇ 후보키 ⊇ 기본키/대체키**, 외래키는 다른 테이블 PK 참조
- **자연키(본질식별자)** = 의미 있는 값 / **대리키(인조식별자)** = 인위적 일련번호
- 자연키가 (1)안 유일 (2)길고 복잡 (3)바뀜 (4)자손 많음 → **인조키**
- PK 기준: NOT NULL · 불변 · 단순 · 대표성
- 인조키를 써도 **자연키는 `UNIQUE`로 보존**

## 정규화

- 요즘엔 그렇게 엄격하지 않지만 이론적으로 알아야 함
- 실제로는 거의 BCMF까지 사용하고 그 이상은 거의 사용하지 않음
- **비정규화** : 아직 정규화를 안한 것
- **반정규화** : 의도적으로 정규화를 위배하는 것 - 성능을 위해서

<br>

### 데이터 중복이 나쁜 이유

- 같은 사실(예: "111은 김수현")이 **여러 행에 반복** 저장됨
- 반복되면 **언젠가 서로 달라진다** → 무엇이 진짜인지 모름
- 저장 공간 낭비는 덤
- 그리고 무엇보다 **이상현상(Anomaly)** 이 발생

<br>

### 이상현상

:  데이터 중복 때문에 테이블을 조작할 때 생기는 비합리적인 현상

<br>

#### 이상현상 (1) 삽입 이상

: 넣고 싶지 않은 데이터까지 억지로 넣거나 원하는 데이터를 넣지 못 하는 문제

- "아직 수강신청은 안 했지만 신규 학생 333을 등록하고 싶다"
- 그런데 이 표는 **과목코드가 있어야 행이 생김** (PK 일부)
- 결국 가짜 과목 정보를 넣거나, 등록 자체가 불가능

| **학번** | **과목코드** | **성명** | **...** |
| --- | --- | --- | --- |
| 333 | ??? | 신입생 | 신청 안 했는데 과목을 넣어야 함 |

<br>

#### 이상현상 (2) 갱신 이상

: 한 사실을 바꾸려면 여러 행을 동시에 고쳐야 하는 문제

학생 111이 개명해서 "김수현 → 김철수"가 됐다고 하면.

| **학번** | **과목코드** | **성명** |
| --- | --- | --- |
| 111 | E | ~~김수현~~ → 김철수 |
| 111 | M | ~~김수현~~ → ??? (빠뜨리면?) |
- 한 행만 고치고 다른 행을 빠뜨리면 → **111이 두 사람**이 됨
- 데이터 정합성 붕괴

<br>

#### 이상현상 (3) 삭제 이상

: 지우고 싶지 않은 정보까지 함께 삭제되는 문제

학생 222가 "과학" 수강을 취소해서 그 행을 지운다면.

| **학번** | **과목코드** | **성명** | **과목명** |
| --- | --- | --- | --- |
| ~~222~~ | ~~S~~ | ~~임영웅~~ | ~~과학~~ |
- 수강 취소만 하려 했는데, **"과학"이라는 과목 정보 자체**가 사라질 수 있음
- 하나의 행에 여러 주제가 섞여 있어서 생기는 일

<br>

### 정규화

- 위 3가지 이상현상의 **공통 원인 = 중복**
- 중복의 원인 = 한 테이블에 **서로 다른 주제의 데이터가 섞여 있음**
- **정규화(Normalization)**: 속성 간 종속 관계를 따져 큰 테이블을 **여러 작은 테이블로 무손실 분해**하는 과정
    - "무손실" = 다시 조인하면 원래대로 복원 가능
    - 보통 외래키를 통해 쪼개짐

| **단계** | **한 줄** | **제거 대상** |
| --- | --- | --- |
| **1NF** | the key | 반복 그룹(다중 값) |
| **2NF** | the whole key | 부분 함수 종속 |
| **3NF** | nothing but the key | 이행 함수 종속 |
| **BCNF** | ...so help me Codd | 후보키가 아닌 결정자 |

<br>

### 함수 종속성 (Functional Dependency)

: X → Y : X를 알면 Y가 하나로 정해진다

- "Y는 X에 종속된다", "X가 Y를 결정한다"
- X = 결정자(Determinant), Y = 종속자(Dependent)

예시:

- **`학번 → 성명`** : 학번을 알면 이름이 정해진다 ✅
- **`과목코드 → 과목명`** : 과목코드를 알면 과목명이 정해진다 ✅
- **`학번 → 과목명`** : 학번만으론 과목명을 못 정한다 ❌

<br>

#### 종속성 3가지

복합키 `{학번, 과목코드}`를 기준으로

| **종류** | **의미** | **예시** |
| --- | --- | --- |
| **완전 함수 종속** | 키 전체가 있어야 정해짐 | {학번, 과목코드} → 평가코드 |
| **부분 함수 종속** | 키의 일부만으로 정해짐 | 학번 → 성명 |
| **이행 함수 종속** | 비(非)키 → 비(非)키로 돌아 정해짐 | 평가코드 → 평가내역 |
- 2NF는 **부분 종속**을 없애고
- 3NF는 **이행 종속**을 없앰

<br>

| **학번** | **성명** | **수강과목** | **과목명** | **평가코드** | **평가내역** |
| --- | --- | --- | --- | --- | --- |
| 111 | 김수현 | E, M | 영어, 수학 | A, C | 최우수, 보통 |
| 222 | 임영웅 | E, S | 영어, 과학 | B, D | 우수, 미흡 |

### 제 1정규형(1NF)

: 모든 속성은 원자값을 가져야 한다 (반복 그룹 제거)

한 칸의 여러 값을 **행으로 풀어** 줌

| **학번** | **과목코드** | **성명** | **과목명** | **평가코드** | **평가내역** |
| --- | --- | --- | --- | --- | --- |
| 111 | E | 김수현 | 영어 | A | 최우수 |
| 111 | M | 김수현 | 수학 | C | 보통 |
| 222 | E | 임영웅 | 영어 | B | 우수 |
| 222 | S | 임영웅 | 과학 | D | 미흡 |
- PK = **{학번, 과목코드}** (복합키)
- 이제 "한 칸 = 한 값" 은 됐지만... 여전히 중복 투성이
    - 성명은 과목코드와 상관없이 학번만으로 정해짐
        - 부분 함수 종속 (`학번 → 성명`) 이 살아있음

<br>

### 제2정규형 (2NF)

: 복합키의 일부에만 종속되는 속성을 없앤다 (부분 종속 제거)

**`{학번, 과목코드}`**에서 종속 관계를 따져보면:

- **`학번 → 성명`** (학번만으로 결정) → **부분 종속, 떼어낸다**
- **`과목코드 → 과목명`** (과목코드만으로 결정) → **부분 종속, 떼어낸다**
- **`{학번, 과목코드} → 평가코드`** → 완전 종속, 남긴다

테이블이 3개가 됨

**학생** (PK: 학번)

| **학번** | **성명** |
| --- | --- |
| 111 | 김수현 |
| 222 | 임영웅 |

**과목** (PK: 과목코드)

| **과목코드** | **과목명** |
| --- | --- |
| E | 영어 |
| M | 수학 |
| S | 과학 |

**수강** (PK: {학번, 과목코드})

| **학번** | **과목코드** | **평가코드** | **평가내역** |
| --- | --- | --- | --- |
| 111 | E | A | 최우수 |
| 111 | M | C | 보통 |
| 222 | E | B | 우수 |
| 222 | S | D | 미흡 |
- **`평가코드 → 평가내역`** : 평가코드만 알면 평가내역이 정해짐
- 그런데 이 둘은 **둘 다 키가 아닌 일반 속성**
- 키가 아닌 속성끼리 종속 = **이행 함수 종속**
    - **`{학번,과목코드} → 평가코드 → 평가내역`**

<br>

### 제3정규형 (3NF)

: 키가 아닌 속성끼리 서로 종속되면 안 된다 (이행 종속 제거)

**`평가코드 → 평가내역`** 을 별도 테이블로 분리

**평가** (PK: 평가코드)

| **평가코드** | **평가내역** |
| --- | --- |
| A | 최우수 |
| B | 우수 |
| C | 보통 |
| D | 미흡 |
- 각 테이블이 **하나의 주제**만 담게 됨

<br>

### BCNF (Boyce-Codd 정규형)

: 모든 결정자(→의 왼쪽)는 후보키여야 한다

**한 교수는 한 과목만** 담당한다고 할 때.

| **학번** | **과목** | **담당교수** |
| --- | --- | --- |
| 111 | 영어 | 김교수 |
| 222 | 영어 | 이교수 |
| 111 | 수학 | 박교수 |
- PK = {학번, 과목}, 종속 관계는 두 가지
    - **`{학번, 과목} → 담당교수`** (완전 종속)
    - **`담당교수 → 과목`** (한 교수 = 한 과목이므로)
- 키가 아닌 속성끼리의 종속은 없으니 3NF는 만족하지만
- 담당교수가 과목을 결정하는 루프가 생김

**3정규형을 만족하면서 BCNF를 위배하려면 무조건 키 쪽으로 루프가 생김**

<br>

- **`담당교수 → 과목`** : 담당교수는 **결정자인데 후보키가 아님** → BCNF 위반
    - 이교수의 담당 과목이 바뀌면 **여러 행**을 고쳐야 함 (갱신 이상)
    - 수강생 없는 교수의 담당 과목은 **저장 불가** (삽입 이상)

**교수** (PK: 담당교수) / **수강** (PK: {학번, 담당교수}) 로 분해:

| **담당교수** | **과목** |
| --- | --- |
| **김교수** | **영어** |
| **이교수** | **영어** |

| **학번** | **담당교수** |
| --- | --- |
| **111** | **김교수** |
| **222** | **이교수** |
- 3NF를 만족하면 **대부분 BCNF도 자동으로 만족**
    - 위반은 "복합키 + 후보키가 아닌 결정자" 조합에서 드물게 등장
    - BCNF를 **"3.5정규형"** 이라고도 부르기도 함
- 단, 위 분해는 **`{학번, 과목} → 담당교수`** 제약을 한 테이블 안에서 강제하지 못하게 됨 — BCNF의 대가

<br>

### 4정규형과 5정규형

실무에서 만나긴 어려움. "이런 것도 있다" 정도만

- **4NF :  다치 종속(MVD) 제거**
    - 서로 **독립적인 1:N 정보 두 개**를 한 테이블에 담지 말 것
    - 예: 학생의 **`동아리`**와 **`보유기술`**을 한 표에 넣으면
        
        모든 조합이 행으로 생김 → 테이블 2개로 분리
        
- **5NF :  조인 종속 제거**
    - **더 쪼개도 무손실 복원**이 되면 끝까지 쪼갠 상태
    - 3개 이상으로 분해해야만 하는 특수한 경우 — 이론적 완성형

<br>

### 정규화의 이점

- **응집도 높은 엔티티 도출**: 관련 속성끼리 모이고 무관한 건 분리됨
- **중복 최소화**: 이상현상이 사라지고 저장 효율도 좋아짐
    - 스토리지 값이 저렴해지면서 중복 최소화는 상대적으로 덜 중요해짐
- **무결성 극대화**: 사실을 한 곳에서만 관리 → 정합성 보장
- **주 식별자 정확성 향상**: 종속성 분석으로 올바른 PK가 드러남
- **확장성 증대**: 업무 변경 시 영향 범위가 명확하고 작음

<br>

### 트레이드오프: 공짜는 아니다

- 테이블이 잘게 쪼개졌다 = 데이터를 다시 합치려면 **조인(JOIN) 필요**
- 학생 이름 + 과목명 + 평가내역을 함께 보려면 4개 테이블 조인
- 읽기가 매우 빈번한 경우, 조인 비용이 부담될 수 있음
- 그래서 **반정규화(의도적 중복 허용)** 가 등장함
- ⇒ 정규화로 "정석"을 만든 뒤, 필요한 곳만 반정규화로 푸는 게 순서
- 비정규화 ≠ 반정규화

<br>

### 실무에서는 어디까지 정규화할까

기준은 하나: **쓰기 정합성이 중요한가, 읽기 성능이 중요한가**

| **영역** | **흔한 수준** | **왜** |
| --- | --- | --- |
| 전통 업무 시스템(OLTP) | 3NF 중심 | 무결성이 최우선, 갱신이 잦음 |
| 금융·공공·대형 ERP | 3NF 엄격 + 일부 BCNF | 감사·규제, 돈이 걸린 정합성 |
| 일반 웹 서비스 | 2NF~3NF 실용 설계 | 개발 속도, 카운터·캐시 컬럼 등 **부분 반정규화** |
| 데이터 웨어하우스(OLAP) | 스타·스노우플레이크 스키마 | 대량 집계·조회, 조인 최소화가 목적 |
| 로그·분석·검색 | 비정규화가 기본 | 수정 없이 쌓기만 함(append-only), 넓은 테이블 |
| NoSQL | 접근 패턴 중심 적극 비정규화 | 조인이 없거나 비쌈, 중복 관리는 앱의 몫 |
- 부분 반정규화 : Counter 같이 집계 하는 것을 따로 캐시DB에 적재하는 경우

<br>

### 실무 : 정규화는 "기본값"

- 갱신이 있는 데이터라면 **3NF로 시작**하는 게 기본값
    - 정규화를 안 한 설계는 되돌리기 어렵지만, 반정규화는 나중에 가능
- 아래로 갈수록 비정규화가 자연스러운 이유: **데이터가 잘 안 바뀌기 때문**
    - 갱신이 없으면 갱신 이상도 없다 → 중복의 비용이 급감
- 같은 데이터도 자리마다 다르게:
    - 주문 DB는 3NF, 분석용 복사본은 스타 스키마, 검색 인덱스는 비정규 문서

<br>

### M:M 관계 해소

- 실세계 업무 대부분은 **M:M 관계**
    - 예: 학생 다수 ↔ 과목 다수
- DB는 M:M을 직접 표현하지 못함
- **관계(중간) 엔티티**를 끼워 넣어 **M:1 + 1:M** 으로 변환

```
학생 (1) ──< 수강 >── (1) 과목
        M:1        1:M
```

- 방금 만든 **수강 테이블**이 바로 그 관계 엔티티

<br>

### 정리

- 중복은 **삽입·갱신·삭제 이상현상**을 만든다
- **함수 종속성(X→Y)**: X를 알면 Y가 정해진다
- **1NF**: 한 칸에 한 값 / **2NF**: 부분 종속 제거 / **3NF**: 이행 종속 제거
- **BCNF**: 모든 결정자는 후보키 — 3NF인데도 남는 중복을 잡는다
- 4NF·5NF는 이론적 완성형 — 실무는 **3NF~BCNF면 충분**
- 한 장짜리 표를 **학생·과목·평가·수강** 4개로 무손실 분해했다
- 정규화는 무결성을 높이지만 **조인이 늘어나는** 트레이드오프가 있다

## 물리 모델링

### 물리 데이터 모델링

: 논리 데이터 모델을 특정 DBMS(MySQL, PostgreSQL, Oracle…) 에 맞는 물리적 요소로 적용하는 과정

- 같은 논리 모델이라도 **DBMS마다** 타입·문법·기능이 다르다
- 그래서 "관계형 데이터베이스에 대한 지식"이 필요하다
- 결과물은 우리가 매일 쓰는 그것 → **테이블, 컬럼, PK, FK, 인덱스**

논리 모델 = **무엇을** 저장할까 / 물리 모델 = **어떻게** 저장할까

<br>

### 물리 모델링 절차

전환은 보통 이 순서로 진행

| **단계** | **하는 일** |
| --- | --- |
| 1. 테이블 매핑 | 엔티티 → 테이블, 단위·우선순위 결정 |
| 2. 컬럼 매핑 | 식별자 → PK, 일반 속성 → 컬럼, 관계 → FK |
| 3. 상세화 | 무결성 · 인덱스 · 반정규화 · 저장기법 |
- 파티셔닝, 샤딩, 클러스터링
    - **파티셔닝** : 테이블 또는 인덱스 데이터를 파티션(Partition) 단위로 나누어 저장하는 것
    - **샤딩** :  각 DB 서버에서 데이터를 분할하여 저장하는 방식
    - **클러스터링** : 여러 데이터베이스 서버를 하나의 시스템처럼 동작하도록 구성

### 1단계 테이블 매핑 - 엔티티를 테이블로

가장 단순한 케이스는 1:1 매핑

- 엔티티 **`회원`** → 테이블 **`MEMBER`**
- 인스턴스(한 명의 회원) → 한 **행(row)**

그런데 **서브타입**이 있으면 고민이 생김

```
[회원]  (슈퍼타입)
  ├─ 개인회원  (서브타입)
  └─ 법인회원  (서브타입)
```

이걸 테이블 **몇 개**로 만들지가 첫 번째 큰 의사결정

<br>

### 서브타입 → 테이블, 3가지 전략

같은 슈퍼/서브타입 구조도 **세 방식**으로 풀 수 있음

| **전략** | **테이블 구성** |
| --- | --- |
| Roll-Up | 하나로 통합 (슈퍼+서브 전부 한 테이블) |
| Roll-Down | 서브타입마다 따로 (슈퍼는 각 테이블에 흡수) |
| Hacksaw | 슈퍼 1개 + 서브 N개 (각각 분리) |

정답은 없음. **어떤 조회가 많은가**로 결정

<br>

#### ① Roll-Up : 하나의 테이블로 통합

슈퍼타입 이름으로 테이블 하나, **구분자 컬럼**으로 서브타입을 나눔

```
CREATE TABLE MEMBER (
  member_id   BIGINT       PRIMARY KEY,
  member_type CHAR(1)      NOT NULL,   -- 'P'개인 / 'C'법인 (구분자)
  name        VARCHAR(50)  NOT NULL,
  birth_date  DATE,                    -- 개인만 사용
  biz_no      VARCHAR(20)              -- 법인만 사용
);
```

서브타입별 전용 컬럼은 **NULL을 허용**할 수밖에 없는 게 특징

<br>

**장단점**

| **장점** | **단점** |
| --- | --- |
| 조회가 간단, 조인이 줄어 속도 유리 | 서브타입별 `NOT NULL`·기본값을 일관되게 못 줌 |
| 복잡한 로직을 하나의 SQL로 처리 | 컬럼이 많아져 저장공간 증가 가능 |
| 서브타입 구분 없이 전체 조회 쉬움 | 항상 구분자로 걸러야 함 |
| 새 서브타입 추가 시 값만 추가 | 일부 서브타입만 봐도 전체 스캔 비용 발생 |

**적용** → 서브타입 간 동질성이 크고, 같이 조회하는 일이 많을 때

<br>

#### ② Roll-Down : 서브타입마다 테이블

슈퍼타입 컬럼을 **각 서브타입 테이블에 복사**해서 따로 만듦

```
CREATE TABLE INDIVIDUAL_MEMBER (
  member_id  BIGINT       PRIMARY KEY,
  name       VARCHAR(50)  NOT NULL,    -- 슈퍼타입 속성
  birth_date DATE         NOT NULL     -- 개인 전용
);
CREATE TABLE CORPORATE_MEMBER (
  member_id  BIGINT       PRIMARY KEY,
  name       VARCHAR(50)  NOT NULL,    -- 슈퍼타입 속성
  biz_no     VARCHAR(20)  NOT NULL     -- 법인 전용
);
```

<br>

**장단점**

| **장점** | **단점** |
| --- | --- |
| 서브타입별 조회 시 구분 불필요 | 둘을 같이 보려면 UNION 필요 → 성능 저하 |
| 가져올 컬럼이 명확 | 통합 SQL 작성이 어려움 |
| `NOT NULL`·기본값을 제대로 지정 가능 | 전체 식별자(UID) 유지 관리 어려움 |
| 테이블 크기가 작아 풀스캔에 유리 | 합친 VIEW는 조회만 가능 |

**적용** → 서브타입별로 따로 처리하는 업무가 많을 때

- 합칠 때 UNION이 필요한데, UNION 쿼리는 성능이 별로 좋지 않음

<br>

#### ③ Hacksaw : 슈퍼 + 서브 각각

공통은 슈퍼타입 테이블에, 전용 컬럼은 서브타입 테이블에. **FK로 연결.**

```
CREATE TABLE MEMBER (              -- 슈퍼타입
  member_id BIGINT PRIMARY KEY,
  name      VARCHAR(50) NOT NULL
);
CREATE TABLE INDIVIDUAL_MEMBER (   -- 서브타입
  member_id  BIGINT PRIMARY KEY,
  birth_date DATE   NOT NULL,
  FOREIGN KEY (member_id) REFERENCES MEMBER(member_id)
);
```

<br>

**장단점**

| **장점** | **단점** |
| --- | --- |
| 통합·분리의 장점을 동시에 수용 | 같이 처리하려면 항상 조인 발생 |
| 슈퍼타입만 조회 시 SQL 간단 | 전체 통합 조회 많으면 과다 조인 |
| 컬럼 중복 없음 → 저장공간 절약 | SQL이 한층 복잡해질 수 있음 |
| 슈퍼/서브 업무 중요도가 대등할 때 적합 | 슈퍼↔서브 데이터 동기화 필요 |

**적용** → 공통(슈퍼) 트랜잭션이 많고, 서브타입 처리도 독립적으로 자주 발생할 때

<br>

#### 어떤 걸 고를까?

일반적이 서비스(ex. 게시판, 쇼핑몰 등)들은 읽기 위주 → 조회가 중요

"실제로 **어떻게 조회**할 건가?"

- 거의 **같이** 조회한다 → **Roll-Up** (조인 없이 한 방에)
- 거의 **따로** 조회한다 → **Roll-Down** (각자 깔끔하게)
- **둘 다** 중요하다, 공통 처리가 많다 → **Hacksaw**

**실무에서 자주 마주치는 의사결정임. "정답"이 아니라 트레이드오프를 이해하는 게 핵심.**

<br>

### 2단계 컬럼 매핑 : 식별자를 키로 전환

논리 모델의 식별자는 물리 모델에서 **키**가 됨

| **논리 모델** | **물리 모델** |
| --- | --- |
| 실질식별자(UID) | Primary Key |
| 보조식별자 | Unique Index |
| 복합 식별자 | 복합 PK (컬럼 순서 검토) |

복합 키는 자주 쓰는 조회 패턴에 맞춰 **앞 컬럼 순서로** 정하면, 인덱스 효율이 좋아진다

<br>

쿼리를 실행할 때는 반드시 실행 계획을 봐야 한다

- MySQL : EXPLAIN

<br>

#### PK vs Unique Index

둘 다 **유일성**을 보장하고 내부적으로 인덱스가 생김. 차이는?

|  | **Primary Key** | **Unique Index** |
| --- | --- | --- |
| **목적** | 제약조건 + 인덱스 | 인덱스 |
| **NULL** | 허용 안 됨 | 허용됨 |
| **개수** | 테이블당 1개 | 여러 개 가능 |
| **식별자 인식** | 가능 (리버스 시) | 불가능 |
| **참조 무결성(FK)** | 지정 가능 | — |

<br>

**결론: 식별자는 PK로!**

**→ 성능 차이는 없다. PK를 쓰는 게 이득.**

- 무결성 오류 방지 — 참조 무결성(FK) 지정 가능
- PK 컬럼엔 **NULL 불가** → 빈 식별자가 원천 차단
- 모델을 되살릴(리버스) 때 식별자를 **명확히** 인식

Unique Index는 "PK는 아니지만 **중복은 막고 싶은**" 보조 식별자에 사용.

<br>

### 3단계 상세화 : 속성을 컬럼으로

이제 한글 속성명을 **실제 컬럼 정의**로 바꿈 (우리나라만 해당)

| **논리 (한글)** | **→** | **물리 (영문)** |
| --- | --- | --- |
| 고객명 | → | `cust_name VARCHAR(50)` |
| 가입일자 | → | `join_date DATE` |
| 가입경로 | → | `join_channel CHAR(3)` |
- **단어 사전**: 한글 → 영문명 통일 (다음 시간 "데이터 표준화"에서!)
    - 한글 5글자 ⇒ VARCHAR(5)
- **도메인 정의**: 타입과 길이 결정 (`VARCHAR(50)`? `INT`? `DECIMAL`?)

<br>

### 컬럼 NULL 허용 여부

NULL을 허용할지는 **데이터의 활용 형태**를 보고 정해야 한다. 왜 중요할까?

```
-- 가입경로가 'A01','B02'가 아닌 '활동' 고객 추출
SELECT cust_id FROM CUSTOMER
WHERE status = '활동'
  AND join_channel NOT IN ('A01', 'B02');
```

**`join_channel`** 이 **NULL이면** 이 행은 결과에서 **빠짐**

(NULL은 **`NOT IN`** 비교에서 참이 되지 않기 때문) **NULL 설계는 버그의 단골 원인**

<br>

### 데이터 무결성 (Integrity)

데이터의 **정확성·일관성·신뢰성**을 잘못된 갱신으로부터 지키는 장치.

| **종류** | **의미** |
| --- | --- |
| 실체 무결성 | 모든 행은 식별자(PK)를 가지고, NULL 없이 유일 |
| 참조 무결성 | FK 값은 부모 PK에 존재하거나 NULL |
| 도메인 무결성 | 타입·길이·`CHECK`·`NOT NULL`·`DEFAULT`·코드값 |
| 사용자 정의 무결성 | 업무 규칙 (트리거 등) |

대부분 우리가 **DDL의 제약조건**으로 직접 표현하는 것들!

<br>

#### 참조 무결성 - FK 삭제 옵션

부모 행이 삭제될 때 자식 행을 어떻게 할지 정함

```
FOREIGN KEY (member_id) REFERENCES MEMBER(member_id)
    ON DELETE CASCADE       -- 부모 삭제 시 자식도 함께 삭제
    -- ON DELETE SET NULL    -- 자식 FK를 NULL로
    -- ON DELETE RESTRICT    -- 자식이 있으면 부모 삭제 거부
```

| **옵션** | **동작** |
| --- | --- |
| `CASCADE` | 부모 삭제 시 자식 행도 삭제 |
| `SET NULL` | 부모 삭제 시 자식 FK를 NULL로 |
| `RESTRICT` | 자식이 있으면 삭제 불가 |

<br>

### 반정규화 (Denormalization) 맛보기

- **정규화**: 중복 제거 → 데이터 깔끔, 변경 안전. 하지만 **조인이 많아짐**
- **반정규화**: 조회 성능을 위해 **일부러 중복을 허용**

**→ 조회 성능을 얻는 대신, 정합성 관리 비용을 떠안는 것.**

같은 데이터가 여러 곳에 있으니, 바뀌면 **모두 같이** 바꿔줘야 함

<br>

#### 반정규화의 종류 (요약)

| **종류** | **아이디어** |
| --- | --- |
| 집계 테이블 | `GROUP BY` 결과를 미리 계산해 저장 (일별 매출 등) |
| 진행 테이블 | 흩어진 상태·진행정보를 별도 테이블로 통합 관리 |
| 테이블 통합 | 자주 같이 쓰는 테이블을 하나로 (1:1 등) |
| 테이블 분할 | 자주 쓰는 컬럼/행만 떼어내기 (수직·수평) |
| 컬럼 반정규화 | 부모/자식에 컬럼을 복사해 조인 제거 |

#### 반정규화 — 작은 예시

주문할 때마다 고객명을 조인해서 가져오기 번거롭다면?

```
CREATE TABLE ORDERS (
  order_id   BIGINT PRIMARY KEY,
  cust_id    BIGINT NOT NULL,
  cust_name  VARCHAR(50),   -- ⚠ CUSTOMER에서 복사 (중복!)
  order_date DATE
);
```

- 장점: 주문 조회 시 `CUSTOMER`와 **조인 불필요**
- 단점: 고객이 개명하면 `CUSTOMER`와 `ORDERS` **둘 다** 갱신해야 함

<br>

#### 반정규화 — 꼭 기억할 것

| **장점** | **단점** |
| --- | --- |
| 조회 성능 향상 (조인 감소) | 변경 성능 저하 (여러 곳 수정) |
| 개발·운영 편의 (데이터가 모여 있음) | 정합성·무결성 저하 (변경 누락 위험) |
- 먼저 인덱스·SQL 튜닝부터 시도하라
**그래도 안 되면, 장단점을 명확히 인식하고 꼭 필요한 만큼만 반정규화.**
- 반정규화는 "성능 최후의 수단"에 가까움

<br>

### 인덱스 설계 (기초)

인덱스 = **데이터를 빨리 찾는 지름길** (책 뒤의 색인).

- **PK·FK엔 인덱스가 자동/관례적으로 생긴다**
- 자주 **조회 조건(WHERE)** 에 쓰이는 컬럼에 인덱스를 고려
- 자주 **같이** 쓰이는 컬럼은 **결합 인덱스** (순서가 중요!)
- 분포도(중복 정도)가 낮은 컬럼일수록 효과적
    - 데이터를 검색했을 때 중복이 전체의 약 20%를 넘기면, 인덱스 안 탐

```
CREATE INDEX idx_orders_cust ON ORDERS (cust_id);
```

- **인덱스는 조회를 빠르게 하지만, 쓰기는 느려짐. 무한정 만들면 안 됨.**

<br>

### 저장기법 · 시스템 속성 (참고)

물리 모델 마지막 단계에서 성능을 위한 저장 전략도 검토.

- **저장기법**: 파티셔닝, 클러스터링, IOT 등
- **저장공간**: 테이블/인덱스 분리 저장, 테이블스페이스 등
- **시스템(공통) 속성**: 모든 테이블에 공통으로 붙이는 관리 컬럼

```
created_at  TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
updated_at  TIMESTAMP,
created_by  VARCHAR(20)
```

이런 감사(audit) 컬럼, 실무에서 정말 자주 봄

<br>

### 정리

- **물리 모델링** = 논리 모델을 특정 DBMS의 **테이블/컬럼/제약**으로 전환
- 서브타입은 **Roll-Up / Roll-Down / Hacksaw** 중 조회 패턴으로 선택
- 식별자 → **PK**(NULL 불가, 1개), 보조식별자 → **Unique Index**
- 속성 → 컬럼: **영문명 · 타입 · NULL · 제약조건** 결정
- **무결성**(실체·참조·도메인)은 대부분 DDL 제약으로 표현
- **반정규화**는 조회 성능 ↔ 정합성 비용의 **트레이드오프**, 신중히
- 인덱스는 PK/FK·자주 쓰는 조회 컬럼에, 과하지 않게
