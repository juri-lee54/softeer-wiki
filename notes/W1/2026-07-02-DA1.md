## 데이터 모델링 개요

### 데이터 모델링

**현실 세계의 업무를, 데이터를 담을테이블 구조로 옮겨 그리는 설계 작업**

- 서비스의 설계도, DB가 잘 동작하도록 만드는 구조
- 무엇을(엔티티) / 어떤 정보를(속성) / 서로 어떻게 연결할지(관계)를 정한다

#### 데이터 모델링을 먼저 하는 이유

- **애매한 부분을 미리 발견**
- 관계자(기획·개발·DBA)가 **같은 그림을 보고 대화**
- 잘못된 가정에서 오는 **위험과 재작업 감소**
- 빠른 의사결정, 빠른 가시화

---

### 3대 핵심 개념

- 엔티티(Entity)   = 업무가 다루는 "어떤 것(Thing)"
    - 업무에서 **관리해야 하는 대상**. 보통 명사
    - 보통 하나의 엔티티가 하나의 **표(테이블)**
    - 판단 기준: **"이걸 따로 저장하고 추적할 필요가 있나?"**
- 속성(Attribute)  = 그것이 가지는 "성격/정보”
    - 엔티티가 가지는 **세부 정보**.
    - 표의 **컬럼(열)**
- 관계(Relationship) = 어떤 것들 "사이의 연결"
    - 엔티티끼리 어떻게 엮이는지.

---

### 도메인 주도 설계(DDD, Domain-Driven Design)

**: 복잡한 비즈니스 로직과 변화하는 요구사항**을 효과적으로 관리하기 위해, 소프트웨어 설계의 중심을 "도메인"에 두는 접근 방식

- 단순히 기술적인 구현에 초점을 맞추는 것이 아니라, **비즈니스 전문가와 개발자가 공통 언어**를 사용하여 문제의 본질을 파악하고, 코드와 모델을 긴밀히 연결하는 데 목적
- 유비쿼터스 랭귀지(Ubiquitous Language) : 도메인 주도 설계(DDD)의 핵심 개념
    - 용어를 단일화

### 데이터 모델링 3단계

#### 개념 모델 - 큰 그림

- 업무를 **굵직한 덩어리**로만
- 세부 기술 무시

```
[회원] ── 주문함 ── [주문] ── 담음 ── [상품]
```

#### 논리 모델 — 규칙과 상세

- 개념을 받아서 **속성·관계·규칙을 또렷하게**.
- **DB에 독립적으로, 업무 규칙을 정확히 표현**

```
[회원] 1 ─── N [주문] 1 ─── N [주문상세] N ─── 1 [상품]
```

#### 물리 모델 — 실제 DB

- 논리 모델을 **특정 DBMS(MySQL 등)에 맞춰** 구현.
    - 테이블명/컬럼명 확정, 타입 지정(**`VARCHAR(50)`**, **`INT`**)
    - **인덱스**, 제약조건, 성능 고려

```
CREATE TABLE member (
  member_id  BIGINT      PRIMARY KEY,
  email      VARCHAR(100) NOT NULL UNIQUE,
  name       VARCHAR(50)  NOT NULL,
  created_at TIMESTAMP    DEFAULT NOW()
);
```

- 데이터 중심 설계이지만 행동도 충분히 고려해야 함
    - 자주 사용할 것 같은 열에 인덱스 등등

## 엔티티와 속성


### 엔티티(Entity) : 인스턴스의 집합

- 업무에서 **관리하거나 처리해야 하는 주요 대상**
- 공통된 특성으로 하나로 묶을 수 있는 **데이터의 집합**
    - 2개 이상의 인스턴스 집합

---
### 엔티티 정의 기준

- 업무에 필요한가? Yes
- 여러 개 모이는가* Yes
- 식별이 되는가? Yes
    - 주문 같은 경우는 식별자(주문ID)로 식별이 안 될 수 있음 - 행동이기 때문에
- 업무에서 실제로 쓰이는가? Yes
- 다른 엔티티와 연결되는가? (관계의 존재) Yes
---

### 엔티티 분류

실무에선 12종으로 잘게 나누지만, 우리는 둘만 기억

- **개체(키) 엔티티** : 그 자체로 존재하는 대상 (명사)
    - 회원, 상품 사원
- **행위(내역) 엔티티** : 무언가 일어난 사건/행위 (동사)
    - 주문, 결제, 로그인이력
    - 행위 엔티티는 발생 시간이 중요
- 개체, 행위 왜 구분?
    - **개체** 엔티티는 먼저 존재한다
    - **행위** 엔티티는 개체들이 만나서 생긴다
    - 행위 엔티티는 보통 **여러 개체를 연결**한다 (회원+상품)
        - (참고) 이 외에도 코드/이력/통계 같은 분류도 있다 — "이런 것도 있구나" 정도면 충분

---
### 슈퍼타입 / 서브타입 구조

```
        회원 (슈퍼타입) — 공통 속성: 이름, 연락처
       ┌──────────┴──────────┐
   개인회원(서브)          법인회원(서브)
   생년월일                사업자번호
```

- **슈퍼타입**: 공통 속성·관계를 보유
- **서브타입**: 자기만의 속성을 추가로 가짐
- **규칙**: 서브타입끼리 **겹치지 않고**, 다 합치면 슈퍼타입과 1:1
    - 모든 회원은 **반드시 하나의** 서브타입에 속한다
    - 애매해서 다 못 나누겠다면 → **'기타'** 구분을 만든다
    - 슈퍼/서브는 **부모-자식 관계가 아니다** (같은 대상을 보는 관점일 뿐)
    - 차이가 거의 없다면 굳이 나누지 말고 그냥 속성 하나로 처리

---
### 속성(Attribute)

- 엔티티 안에서 **관리하고 싶은 정보 항목 하나하나**
- 그 대상의 **본질적인 성질**
- 속성도 결국 값들의 집합

---

### 좋은 속성의 조건

- **원자값**
    - 원칙: **일단 최소 단위로 쪼갠 뒤, 필요에 따라 합친다**
    - **일자·시간·성명·주민번호·우편번호**는 보통 쪼개지 않는 게 편함
    - **주소**는 일단 두었다가 설계 때 필요하면 쪼갬
    - 정답은 **업무 요구사항**이 결정한다
- **단일값**
    - 하나의 인스턴스에 대해 그 속성은 **값을 딱 하나만** 가져야 함
    - 반복되는 값은 → **별도 엔티티로 분리** (제1정규화)
        - **자식 엔티티로 빼낸다 !!!!!!!!!**
    - 요즘은 JSON도 저장하는 추세

---

### 속성 검증 4단계

| **단계** | **무엇을 확인** |
| --- | --- |
| **1. 최소 단위 분할** | **원자 단위까지 쪼갰나? (목적 고려)** |
| **2. 유일값 검증** | **값이 하나뿐인가? 반복되면 분리** |
| **3. 가공값 판정** | **계산으로 다시 만들 수 있는 값인가?** |
| **4. 관리수준 검토** | **더 상세히 관리할 필요가 있나?** |
- 3번: 합계·나이처럼 **재현 가능한 값**은 논리모델에선 보통 생략
- 집계 하는 count 쿼리는 캐시DB를 사용해서 따로 구함

---

### 필수 속성 vs 선택 속성

논리적 단계에서 수행

- **필수(Mandatory)**: 값이 **반드시** 있어야 함 — 표시 `*`
- **선택(Optional)**: 값이 없어도 됨 — 표시 **`o`**

| **속성** | **회원번호** | **이름** | **이메일** | **가입일** | **직업** |
| --- | --- | --- | --- | --- | --- |
| **구분** | * | * | * | * | o |
- 필수/ 선택을 정하는 것도 모델링의 일부 - 업무 규칙으로 결정

## 관계와 ERD

### 관계(Relationship) = 엔티티 사이의 업무적 연관성

- DB에서는 **참조 무결성(referential integrity)** 을 지켜주는 핵심 장치
    - 존재하지 않는 고객의 주문은 만들 수 없다
    - 외래키 = 참조무결성 ?
- **업무 규칙**을 담는다
- **관계는 3가지로 표현한다: 관계명, 기수성(Cardinality), 선택성(Optionality)**
- 두 엔티티에 관계가 있다 = **부모의 식별자(PK)**를 자식이 **관계 속성(FK)**로 물려 받는다.
    - 관계 = 코드 레벨에서는 결국 **외래키 컬럼** 으로 구현된다

#### 기수성

- 한 엔티티의 인스턴스 하나가 상대 쪽 인스턴스를 몇 개 가질 수 있는가?
- 1:1, 1:N, M:N
- 대부분의 실무 관계는 **1:N** 으로 정리된다.
- M:N은 어플리케이션 설계 관점으로도 좋지 않기 때문에 없애는 게 좋음

#### 선택성

- **관계가 반드시 있어야 하는가, 아니면 없을 수도 있는가?**
- 필수 (Mandatory), 선택 (Optional)
- 기수성과 선택성은 **방향마다 따로** 본다
    - "고객→주문"과 "주문→고객"의 선택성은 다를 수 있다

---

### 한 관계를 양방향으로 읽기

예: **고객 — 주문**

- 고객 입장: "한 고객은 **0개 이상**의 주문을 가질 수 있다" (선택, 다)
- 주문 입장: "한 주문은 **반드시 한** 고객에 속한다" (필수, 하나)

이렇게 **양쪽 방향의 기수성 + 선택성** 을 함께 적으면 관계가 완성된다.

물리적 설계 - **테이블**

### 식별 관계 vs 비식별 관계

- 식별 관계는 굉장히 드뭄.
- 대부분의 관계는 비식별(점선)

|  | **식별 관계** | **비식별 관계** |
| --- | --- | --- |
| **FK 위치** | **자식 PK의 일부** | **자식 일반 컬럼** |
| **존재 종속** | **O (부모 없이 못 태어남)** | **O** |
| **식별 종속** | **O** | **X** |
| **까마귀발 선** | **실선** | **점선** |

---

### 1:1 관계

- 실무에서는 **드물게** 발생
- 1:1이 자주 보인다면 → **사실 같은 엔티티** 일 가능성이 높다
    - 굳이 나눌 이유가 있을 때만 분리
        - 자주 안 쓰는 큰 컬럼 분리
            - 데이터베이스의 물리적인 접근 단위는 레코드.
        - 접근 권한 분리

### M:1 (1:N) 관계 — 가장 일반적

- 현실 세계는 수평 관계보다 **수직(상하) 관계** 가 더 많다
- 자식(사원)은 부모(부서) 없이 함부로 태어날 수 없고,
    
    부모(부서)는 자식(사원)이 없을 수도 있다
    

### M:N 관계

- 꽤 자주 발생하지만 — **아직 덜 풀린 상태**
- 관계형 DB는 M:N을 **직접 표현할 수 없다**
    - FK 하나로 양쪽을 다 가리킬 수 없기 때문
- 그래서 논리적 설계 단계에서는 **교차 엔티티(중간 테이블)** 로 풀어줘야 한다
    - **M:N은 "언제" 풀까?**
        - 관계 자체에 **속성** 을 채워야 할 때 (수량, 가격, 신청일...)
        - 관계가 또 다른 **자식 엔티티** 를 가져야 할 때
        - 관계명을 못 붙이겠다면, 그건 **이름 없는 엔티티** 가 숨어 있다는 뜻
        - **복잡성에 빠지지 말 것 — 필요할 때 교차 엔티티로 단순하게 푼다**

---

### 다중관계 해소 — 두 엔티티가 여러 번 얽힐 때

예: **주문** 에는 여러 종류의 고객이 붙는다

(주문고객, 결제고객, 수령고객 ...)

두 가지 해소 방법:

- **병렬식**: FK 컬럼을 여러 개 늘린다
    - 컬럼으로 관리
- **직렬식**: 관계 엔티티를 따로 둔다
    - 로우로 관리

---
### 병렬식 vs 직렬식

|  | **병렬식 (컬럼 추가)** | **직렬식 (관계 엔티티)** |
| --- | --- | --- |
| **방식** | **주문에 FK 여러 개** | **주문-고객역할 테이블 신설** |
| **엔티티 수** | **안 늘어남** | **하나 늘어남** |
| **관계 추가 시** | **컬럼이 계속 증가** | **로우(행) 만 추가** |
| **이력 관리** | **어렵다** | **쉽다** |
| **유연성** | **낮음 (전용 관계)** | **높음** |
- 종류가 **고정·소수** → 병렬식이 단순해서 OK
- 종류가 **늘어나거나 이력**이 필요 → 직렬식 (M:N 푸는 것과 같은 발상)

### 실선 vs 점선

- **실선** = 식별 관계 (FK가 자식 PK의 일부)
- **점선** = 비식별 관계 (FK가 자식 일반 컬럼)

대부분의 관계는 **점선(비식별)**.

실선은 "이 자식은 부모 없이는 식별조차 안 된다"는 강한 결합일 때만.

### 실습 — 상품 주문 과정을 ERD로

**예제의 엔티티를 정의하고, ERD를 그려 보자**

1. 고객은 여러 **배송지**를 등록해 두고 상품 주문 시 선택할 수 있다.
    
    가입 시 주소가 기본 배송지가 되며, 기본 배송지는 변경할 수 있다.
    
2. 고객은 여러 **상품**을 하나의 **주문**으로 묶을 수 있다.
3. 하나의 주문은 신용카드·포인트 등 여러 **결제수단**으로 결제할 수 있다.
