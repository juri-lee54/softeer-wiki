# 미션 설명
이 미션이 하는 일

```
[내 컴퓨터 (호스트)]
   │
   │  docker build / docker run
   ▼
[Docker 컨테이너 = 가상의 리눅스 서버 1대]
   │
   ├─ NameNode          ┐
   ├─ DataNode          ├─ HDFS (분산 파일 저장 시스템)
   ├─ SecondaryNameNode ┘
   │
   ├─ ResourceManager   ┐
   └─ NodeManager       ┴─ YARN (자원 관리 / 작업 실행 시스템)

```

Hadoop은 원래 수십~수천 대의 서버에 나눠 돌아가는 분산 시스템. 이 미션은 그 전체
클러스터를 컨테이너 1개 안에 압축해서, "가짜 분산 모드(pseudo-distributed mode)"로
돌리는 것. 서버가 1대뿐이지만 Hadoop 입장에서는 여전히 "NameNode 서버", "DataNode 서버"가 따로 있는 것처럼 동작, 다만 그 서버들의 주소가 전부 localhost.

<br>


# 개념
## Hadoop 핵심 개념

### HDFS (Hadoop Distributed File System)
- 하나의 큰 파일을 여러 대의 서버에 **블록(block) 단위**(기본 128MB)로 쪼개어 분산 저장하는 파일시스템.
- 일반 리눅스 파일시스템과 별개의 "가상 파일시스템"이며, 전용 명령어(`hdfs dfs`)로만 접근
- 컨테이너의 로컬 디스크(`/tmp`, `/home` 등)와 HDFS(`/user/root/...`)는 **완전히 다른 저장 공간**이라는 점이 중요.

<br>

### HDFS를 구성하는 데몬

| 데몬 | 역할 |
|---|---|
| **NameNode** | HDFS의 "관제탑". 어떤 파일이 어떤 블록들로 쪼개져 있고, 그 블록이 어느 DataNode에 있는지 메타데이터로 관리. NameNode가 죽으면 HDFS 전체를 못 씀 |
| **DataNode** | 실제 파일 블록(바이트 데이터)을 디스크에 저장하는 워커. 여러 대 있을수록 저장 용량과 처리량이 늘어남 |
| **SecondaryNameNode** | NameNode의 백업이 아니라, 메타데이터 편집 로그를 주기적으로 병합(체크포인트)해서 NameNode의 부담을 줄여주는 보조 프로세스 |

<br>

### YARN (Yet Another Resource Negotiator)
Hadoop에서 실제 연산 작업(MapReduce, Spark 등)에 CPU/메모리 자원을 배분하고 스케줄링하는 시스템.

| 데몬 | 역할 |
|---|---|
| **ResourceManager** | 클러스터 전체 자원을 관리하는 마스터. "이 작업에 컨테이너(자원 단위) 몇 개를 줄지" 결정 |
| **NodeManager** | 각 서버에서 실제로 작업 프로세스를 실행/모니터링하는 워커 |

<br>


### 리플리케이션(Replication)
HDFS는 데이터 안정성을 위해 같은 블록을 여러 DataNode에 복제해서 저장함.

기본값은 3벌 복제이지만, 이 미션처럼 DataNode가 1대뿐이면 복제본을 3개 만들 수 없으므로
`dfs.replication=1`로 낮춰야 함

<br>

### 네임노드 포맷(Format)
`hdfs namenode -format`은 HDFS를 "초기화"하는 명령. 

빈 메타데이터(클러스터 ID, 빈 파일 트리)를 새로 만듦

**한 번 포맷된 클러스터를 다시 포맷하면 기존 메타데이터가 전부 사라지므로, 볼륨에 이미 데이터가 있다면 재포맷하면 안 됨** 


<br>

### SSH가 왜 필요한가
Hadoop의 `start-dfs.sh` / `start-yarn.sh` 스크립트는 원래 여러 대의 서버에 SSH로 접속해서 각 서버의 데몬을 원격으로 켜도록 설계되어 있음. 

단일 노드라도 이 로직이 그대로 남아있어서, **자기 자신(localhost)에게 비밀번호 없이 SSH 접속**할 수 있어야 정상 동작.

그래서 Dockerfile에서 SSH 키를 만들고 `authorized_keys`에 자기 공개키를 등록해둔 것입니다.


<br>

## Hadoop 설정 파일 (XML)

Hadoop은 자바 프로그램이며, 시작할 때 `$HADOOP_HOME/etc/hadoop/*.xml`을 자동으로 읽어 설정값으로 사용하도록 만들어져 있음. 

**정해진 경로의 XML 파일** = 설정 


| 파일 | 담당 영역 | 이 미션에서 지정한 핵심 값 |
|---|---|---|
| `core-site.xml` | Hadoop 전체 공통 설정 | `fs.defaultFS=hdfs://localhost:9000` (기본 파일시스템을 HDFS로 지정) |
| `hdfs-site.xml` | HDFS 저장 계층 설정 | `dfs.replication=1`, `name.dir`/`data.dir` 저장 경로, `webhdfs.enabled=true` |
| `mapred-site.xml` | MapReduce 실행 방식 | `mapreduce.framework.name=yarn` |
| `yarn-site.xml` | YARN 자원 관리 | `yarn.nodemanager.aux-services=mapreduce_shuffle` |
| `hadoop-env.sh` | 셸 환경변수 | `JAVA_HOME` 지정 (Hadoop 스크립트가 자바 위치를 찾는 데 사용) |

기본값을 그대로 두면: HDFS가 활성화되지 않거나(`file:///`가 기본 파일시스템), 복제본 3개를
요구해서 경고가 뜨거나, 데이터가 `/tmp`(임시 공간)에 저장되어 재시작 시 사라짐.
즉 XML 설정은 "Hadoop의 기본 동작을 우리의 단일 노드 + 영속화 환경에 맞게 오버라이드하는 것"


<br>

## Hadoop 실행/관리 스크립트 & 명령어

| 명령어 | 하는 일 |
|---|---|
| `hdfs namenode -format` | HDFS 메타데이터 초기화 (최초 1회만) |
| `start-dfs.sh` | NameNode, DataNode, SecondaryNameNode 기동 |
| `start-yarn.sh` | ResourceManager, NodeManager 기동 |
| `stop-dfs.sh` / `stop-yarn.sh` | 위 데몬들을 정지 |
| `jps` | 현재 실행 중인 자바 프로세스 목록 확인 (Hadoop 데몬이 다 떠 있는지 검증용) |

`jps` 실행 시 정상이라면 아래가 모두 보여야 함
```
NameNode
DataNode
SecondaryNameNode
ResourceManager
NodeManager
```

<br>

## HDFS 조작 명령어 (`hdfs dfs`)

`hdfs dfs`는 리눅스 명령어(`ls`, `mkdir`, `cp`, `cat`, `rm`)를 HDFS용으로 그대로 옮긴 것입니다.

| 명령어 | 리눅스 유사 명령 | 하는 일 |
|---|---|---|
| `hdfs dfs -mkdir -p /user/root/mydata` | `mkdir -p` | HDFS 안에 디렉토리 생성 |
| `hdfs dfs -ls /user/root/mydata` | `ls` | HDFS 디렉토리 내용 목록 확인 |
| `hdfs dfs -put <로컬경로> <HDFS경로>` | `cp` (업로드 방향) | 로컬 파일을 HDFS로 업로드 |
| `hdfs dfs -get <HDFS경로> <로컬경로>` | `cp` (다운로드 방향) | HDFS 파일을 로컬로 다운로드 |
| `hdfs dfs -cat <HDFS경로>` | `cat` | HDFS 파일 내용을 터미널에 바로 출력 |
| `hdfs dfs -rm <HDFS경로>` | `rm` | HDFS 파일 삭제 |
| `hdfs dfs -df -h` | `df -h` | HDFS 전체 용량/사용량 확인 |
| `hdfs dfs -du -h <경로>` | `du -h` | 특정 경로의 용량 확인 |

`-put`과 `-get`은 서로 반대 방향. 

`-put`은 "로컬 → HDFS", `-get`은 "HDFS → 로컬"이라고 외우면 헷갈리지 않음

|---|---|
