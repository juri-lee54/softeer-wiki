# 미션 설명

MovieLens 20M 데이터셋(`ratings.csv`, 약 2천만 건)을 Hadoop Streaming + Python으로 처리해, 영화(movieId)별 **평균 평점**을 계산하는 미션. 클러스터·제출 방식은 w3m3/w3m4와 동일하고, 이번에 새로 등장하는 건 **Reducer가 "개수 세기"가 아니라 "평균 내기"를 해야 한다는 점**과 **숫자형 CSV 데이터를 다루는 것**이다.

# 새로 배운 핵심 개념

## 1. 헤더/파싱 실패를 하나의 예외처리로 묶어서 거르기

`ratings.csv`는 첫 줄에 `userId,movieId,rating,timestamp` 헤더가 있다. w3m3(전자책)는 헤더가 없었고 w3m4(Sentiment140)도 헤더 없는 데이터였어서, **"첫 줄만 따로 건너뛰기"를 명시적으로 처리해본 적이 없었다.**

이번엔 별도 분기 없이, `rating` 자리 값을 `float()`로 변환하는 과정에서 자연스럽게 걸러지도록 짰다.

```python
try:
    movie_id = fields[1]
    rating = float(fields[2])
except (IndexError, ValueError):
    continue
```

헤더 줄의 `rating` 자리엔 문자열 `"rating"`이 들어있어서 `float("rating")`이 `ValueError`를 낸다. 즉 "잘못된 줄 방어용 예외처리"가 "헤더 스킵"까지 덤으로 해결해준 것 — 예외 상황을 먼저 정의하고 나니 별도 로직 없이도 처리가 된 케이스.

<br>

## 2. `csv` 모듈이 필요 없는 경우도 있다

w3m4에서는 트윗 본문에 콤마가 그대로 섞여 있어서 `csv.reader`가 필수였다. 이번 `ratings.csv`는 전부 **숫자 필드**(`userId,movieId,rating,timestamp`)라 필드 안에 콤마가 들어갈 일이 없다. 그래서 `line.split(",")`만으로 충분했다.

> => `csv` 모듈은 "구분자가 필드 내용에도 나타날 수 있는 데이터"를 다룰 때 필요한 것이지, "콤마로 구분된 데이터"라고 무조건 필요한 게 아니다. 데이터의 실제 구조(전부 숫자인지, 자유 텍스트가 섞여있는지)를 보고 판단해야 한다.

<br>

## 3. Reducer가 "개수"가 아니라 "평균"을 낼 때 — sum과 count를 동시에 추적

w3m3(단어 수), w3m4(카테고리 개수)는 Reducer가 값을 그냥 **더하기만** 하면 끝이었다. 이번엔 평균을 내야 해서 **합계(`sum_rating`)와 개수(`count`) 두 개를 같이 누적**하고, key가 바뀌는 시점에 `sum / count`를 계산해서 출력해야 한다.

```python
if movie_id != current_movie:
    if current_movie is not None:
        print(f"{current_movie}\t{sum_rating / count:.1f}")
    current_movie, sum_rating, count = movie_id, 0.0, 0
sum_rating += rating
count += 1
```

그룹 전환/flush 뼈대 자체는 w3m3와 완전히 같지만(→ 겹치는 개념 참고), **누적하는 상태가 값 하나(count)에서 값 두 개(sum, count)로 늘었다는 점**이 이번에 새로 신경 써야 했던 부분이다. 카운트만 있을 땐 초기값 0에 정수를 더하기만 하면 됐지만, 평균은 나눗셈이 들어가므로 두 상태를 항상 짝지어 리셋해야 한다.

<br>

## 4. 반올림 — `round()` vs f-string 포맷의 차이

과제 스펙이 "소수점 첫째 자리까지" 출력하라는 요구였는데, 반올림하는 두 가지 방법의 차이를 정리해둔다.

- `round(x, 1)`: 결과가 `float` 객체. 부동소수점 특성상 드물게 `3.4999999999999996`처럼 깨끗하지 않은 값이 나올 수 있고, 정수 평균(`4.0`)일 때 그대로 `4.0`으로 나오긴 하지만 항상 소수점 자리가 보장되는 건 아니다.
- `f"{x:.1f}"`: **문자열 포맷팅**이라 항상 소수 첫째 자리까지 고정된 형식(`4.0`, `3.5`)으로 출력된다.

출력 형식의 일관성이 중요한 과제(스펙에 `3.5`, `4.0` 같은 예시가 명시됨)에서는 `:.1f` 포맷 방식이 더 안전하다.

<br>

## 5. Hadoop Streaming의 기본 정렬은 key를 "문자열"로 취급한다

로컬 `sort`와 Hadoop Streaming의 셔플 정렬 모두, movieId를 숫자가 아니라 **문자열 사전식**으로 정렬한다. 그래서 결과가 `1, 10, 100, 1000, 100006, ..., 112, 1136, ...` 순서로 나온다 (`112`가 `1097`보다 뒤에 오는 이유).

> => 이건 버그가 아니라 기본 동작이다. 같은 movieId끼리만 뭉쳐있으면 Reducer의 group-by 로직엔 전혀 영향이 없다. 다만 "출력이 숫자 순서로 안 나오는데 잘못된 거 아니냐"고 오해하기 쉬운 부분이라 별도로 정리해둔다. (숫자 순 정렬이 꼭 필요하면 `mapreduce.job.output.key.comparator.class`로 별도 Comparator를 지정해야 하는데, 이번 과제 스펙은 순서를 요구하지 않아서 그대로 두었다.)

<br>

# 트러블슈팅

## Reducer에서 문자열을 그대로 합산하려다 난 `TypeError`

```python
def parse(self, line):
    movie_id, rating = line.strip().split('\t', 1)
    return (movie_id, rating)     # rating이 여전히 문자열
```

`split('\t', 1)`의 결과는 항상 문자열이다. `sum_rating`(`float` 타입, 초기값 `0.0`)에 문자열 `rating`을 그대로 더하면:

```
TypeError: unsupported operand type(s) for +=: 'float' and 'str'
```

Mapper에서는 `float(fields[2])`로 변환했지만, Reducer의 `parse()`에서 그 변환을 또 한 번 해줘야 한다는 걸 깜빡한 것. w3m4 트러블슈팅에 정리된 "문자열 vs 정수" 실수와 같은 종류지만, 그땐 `int()` 누락이었고 이번엔 `float()` 누락이었다 — **탭으로 split한 값은 항상 문자열이므로, Mapper와 Reducer 양쪽에서 각각 형변환이 필요하다**는 걸 매번 확인해야 한다.

```python
return (movie_id, float(rating))   # 수정
```
