# 미션 설명

Amazon Reviews 2023 데이터셋(`Movies_and_TV` 카테고리, 약 1,700만 건)을 Hadoop Streaming + Python으로 처리해, 제품(`parent_asin`)별 **리뷰 수와 평균 평점**을 함께 계산하고 가장 많은 리뷰를 받은 제품을 찾는 미션. 클러스터·제출 방식·Mapper/Reducer의 기본 골격은 w3m5(MovieLens 평균 평점)와 거의 동일하고, 새로 신경 쓴 부분은 **출력에 개수까지 포함시키는 것**과 **어떤 데이터셋 파일을 받아올지 스스로 골라야 했다는 것**이다.

<br>

# 새로 배운 핵심 개념

## 1. Reducer가 "평균"뿐 아니라 "개수"도 같이 출력해야 하는 경우

w3m5는 그룹별로 `sum_rating`과 `count`를 같이 누적하긴 했지만, 최종 출력엔 평균만 내보내고 `count`는 계산에만 쓰고 버렸다. 이번 과제 스펙은 `제품ID, 리뷰 수, 평균 평점`을 전부 요구해서, flush 시점에 `count`도 그대로 같이 출력하면 된다.

```python
def emit(self, product_id, sum_rating, count):
    print(f"{product_id}\t{count}\t{round(sum_rating / count, 1)}")
```

누적 로직 자체는 w3m5와 완전히 같고, **"이미 계산에 쓰던 중간값을 출력에도 재사용할 수 있는지"만 확인하면 되는 문제**였다. 새로 추적해야 할 상태가 늘어난 게 아니라, 이미 갖고 있던 값을 버리지 않고 내보내기만 하면 됐다.

<br>

## 2. 거대한 공개 데이터셋은 "필요한 컬럼만 있는 축소판"부터 찾아본다

Amazon Reviews 2023은 카테고리별로 리뷰 원문 + 메타데이터가 담긴 `.jsonl.gz`(수백 MB~수 GB, JSON 파싱 필요)를 기본 제공하지만, 문서를 더 보면 추천 시스템 벤치마크용으로 **"Pure IDs (rating only)"** 라는, `user_id,parent_asin,rating,timestamp` 네 컬럼만 남긴 압축 CSV를 별도로 제공한다.

이번 과제는 제품ID·평점만 필요해서 리뷰 원문이나 메타데이터가 전혀 필요 없었다. Pure IDs 버전을 쓰니 JSON 파싱 없이 MovieLens `ratings.csv`와 똑같은 형태(`csv` 4컬럼, 숫자/ID 위주)로 바로 처리할 수 있었다.


<br>

## 3. 외부에 알려진 정답이 없을 때 — 로컬 셸 파이프라인으로 직접 ground truth를 만든다

w3m5는 `movieId 296`(Pulp Fiction)의 실제 평균 평점이 공개적으로 알려져 있어서, 그 값과 비교하는 것으로 검증이 끝났다. 이번엔 어떤 `parent_asin`이 1등인지, 그 값이 맞는지 확인해줄 외부 자료가 없었다.

그래서 Hadoop에 던지기 전에 로컬에서 전체 데이터로 직접 정답을 만들어뒀다.

```bash
tail -n +2 sample/Movies_and_TV.csv | cut -d',' -f2 | sort | uniq -c | sort -rn | head -5
```

이 결과(최다 리뷰 제품과 건수)를 미리 확보해두고, Hadoop 결과와 정확히 일치하는지 대조하는 방식으로 검증했다.

"샘플 100줄로 로직만 확인"(w3m3~w3m5에서 하던 것)과 "전체 데이터로 직접 정답을 만들어 최종 결과와 대조"는 다른 단계의 검증이다. 외부에 검증 가능한 사실이 없는 데이터셋일수록, 셸 도구(`cut`/`sort`/`uniq -c`)로 만든 로컬 집계가 신뢰 기준이 된다.

<br>

## 4. 이전 미션 폴더를 복사해 새 미션을 시작하면, 포맷된 HDFS 상태까지 같이 복사된다

W3M6 폴더는 W3M5를 통째로 복사해서 시작했다. `entrypoint.sh`는 `NAMENODE_DIR/current` 디렉토리가 있으면 "이미 포맷됨"으로 판단하고 포맷을 건너뛰는데(→ [w3m1.md](./w3m1.md) 재포맷 주의사항), 복사해온 `data/master/name/current`가 그대로 남아있어서 **새 미션인데도 예전 클러스터(다른 데이터셋으로 포맷된) 상태를 그대로 이어받는** 상황이 됐다.

```bash
rm -rf W3M6/data   # 복사돼온 stale 상태 제거 → 다음 up에서 새로 포맷됨
```

> => "재포맷하면 안 된다"는 원칙은 *같은* 클러스터를 이어 쓸 때의 얘기고, 폴더 자체를 복사해서 **새 미션(새 데이터셋)을 시작**하는 경우엔 반대로 `data/`를 지우고 완전히 새로 포맷돼야 한다. 폴더를 템플릿처럼 복사해 쓸 땐 `sample/`(입력 데이터)뿐 아니라 `data/`(클러스터 상태)도 같이 따라온다는 걸 매번 확인해야 한다.

# 트러블슈팅

## `docker compose up`이 컨테이너 이름 충돌

모든 미션 폴더의 `docker-compose.yml`이 `container_name: master/worker1/worker2`를 그대로 쓰고 포트(`9870`/`8088`/`9000`)도 동일하게 매핑한다. W3M5 클러스터가 떠 있는 상태에서 W3M6을 `up` 했으면 컨테이너 이름·포트 충돌로 바로 실패했을 것 

-  실행 전에 `docker ps`로 이전 미션(W3M5) 컨테이너가 떠 있는 걸 확인하고, `docker compose stop`으로 먼저 내린 뒤 W3M6을 올렸다.

> => 미션마다 폴더는 분리돼 있지만 컨테이너 이름/포트는 전부 재사용하는 구조이기 때문에, **한 번에 하나의 미션 클러스터만 띄울 수 있다.** 새 미션을 시작하기 전엔 `docker ps`로 이전 컨테이너 상태부터 확인하는 습관이 필요하다.
