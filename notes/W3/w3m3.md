# 미션 설명

지금까지 만든 멀티노드 Hadoop 클러스터(master + worker1 + worker2) 위에서, **실제 MapReduce 작업(단어 수 세기)을 Python으로 작성하고 실행**하는 미션. 대상 데이터는 전자책 한 권(Designing Data-Intensive Applications, Martin Kleppmann) 전체 텍스트.

```
전자책(PDF) → pdftotext → HDFS 업로드 → Hadoop Streaming으로 Python mapper/reducer 실행 → HDFS에 결과 저장
```

w3m1/w3m2a/w3m2b가 "클러스터를 구성하고 설정을 검증하는 것"이었다면, w3m3는 그 위에서 **실제 분산 연산 작업을 처음부터 끝까지 돌려보는 것**이 핵심.

# 새로 배운 핵심 개념

## 1. 입력 데이터는 반드시 "줄 단위 텍스트"여야 한다

전자책이 PDF로만 있었는데, Hadoop Streaming의 기본 입력 방식은 파일을 **줄(line) 단위**로 읽어 mapper의 stdin에 흘려보내는 것이다. PDF는 내부적으로 압축된 바이너리 포맷이라 그대로 넣으면 단어가 아니라 깨진 바이트가 나온다.

```bash
pdftotext ./sample/DDIA.pdf ./sample/DDIA.txt   # PDF -> plain text 변환을 먼저 해야 함
```

> => MapReduce에 넣기 전에 **입력 포맷이 실제로 줄 단위 텍스트인지**부터 확인해야 한다. HDFS는 파일을 그저 바이트 블록으로 나눠 저장할 뿐, 포맷을 해석해주지 않는다.

<br>

## 2. Hadoop Streaming — Java 없이 아무 언어로나 mapper/reducer 작성

Hadoop은 원래 Java로 `Mapper`/`Reducer` 클래스를 상속받아 작성하는 게 기본이지만, **Hadoop Streaming**이라는 유틸리티(jar)를 쓰면 실행 파일(여기선 Python 스크립트)을 그대로 mapper/reducer로 꽂아 쓸 수 있다.

핵심 동작 방식:
- Hadoop이 입력을 한 줄씩 mapper 프로그램의 **stdin**에 넣어준다
- mapper가 **stdout**으로 내놓는 `key\tvalue` 형식의 줄을 mapper 출력으로 취급한다
- reducer도 마찬가지로 stdin/stdout 기반으로 동작한다

즉 Java의 `Mapper<K,V>`/`Reducer<K,V>` 인터페이스가 stdin/stdout 파이프로 대체된 것.

<br>

## 3. "Mapper/Reducer 클래스로 구현하라"는 요구사항 해석 — mrjob vs 순수 Streaming

과제 요구사항 문구가 "Mapper 및 Reducer **클래스**를 Python으로 구현"이라 두 가지 선택지가 있었다.

| 방식 | 특징 |
|---|---|
| **mrjob 라이브러리** | `class MRWordCount(MRJob):` 형태로 정의, `mapper()`/`reducer()`를 메서드로 작성. job 제출 자체도 라이브러리가 대신 조립해줌 |
| **순수 Hadoop Streaming + 클래스 래핑** | `mapper.py`/`reducer.py`를 그대로 쓰되, 로직을 `class WordMapper:`/`class WordReducer:`로 감싸서 요구사항 문구를 만족 |

실제로는 **mrjob을 선택했다가 포기하고 순수 Streaming 방식으로 전환**했다. 이유는 아래 Dockerfile 문제 참고.

> => `mrjob`은 **PyPI(pip) 패키지**이지 Ubuntu apt 패키지가 아니다. Dockerfile의 `apt-get install -y ... mrjob ...` 목록에 넣으면 `E: Unable to locate package mrjob`로 빌드 자체가 깨진다. Python 라이브러리는 apt가 아니라 `pip3 install` 로 설치해야 한다는, 기본적이지만 놓치기 쉬운 구분.

<br>

## 4. Streaming Reducer는 "그룹핑"을 직접 구현해야 한다

Java MapReduce API는 reducer에 `(key, Iterable<values>)`로 이미 그룹핑된 데이터를 넘겨주지만, **Hadoop Streaming의 reducer는 정렬된 `key\tvalue` 줄들을 그냥 스트림으로 흘려보낼 뿐**이다. 같은 키가 연속되는 구간을 reducer 스크립트가 직접 추적해야 한다.

```python
current_word = None
current_count = 0

for line in sys.stdin:
    word, count = parse(line)
    if word == current_word:
        current_count += count          # 같은 키가 계속되면 누적
    else:
        if current_word is not None:
            print(f"{current_word}\t{current_count}")   # 키가 바뀌는 순간 이전 키 결과를 출력
        current_word, current_count = word, count

if current_word is not None:
    print(f"{current_word}\t{current_count}")   # 루프가 끝난 뒤 "마지막 키"를 한 번 더 출력해야 함
```

마지막 `if` 블록을 빼먹으면 **가장 마지막 단어의 카운트가 통째로 누락**된다. 루프 안의 출력 로직은 "다음 키로 넘어갈 때"만 실행되므로, 스트림이 끝나는 시점엔 아무도 마지막 키를 출력해주지 않기 때문.

<br>

## 5. 작은 입력 파일에서 분산 처리를 실제로 끌어내는 법

DDIA.txt처럼 파일이 HDFS 블록 크기(기본 128MB)보다 훨씬 작으면, 기본 설정으로는 **split이 1개 → mapper가 1개만** 뜬다. worker가 2대여도 사실상 혼자 다 처리하고 끝나버림. 이를 강제로 병렬화하려면 job 제출 시 옵션을 추가해야 한다.

```bash
hadoop jar $HADOOP_HOME/share/hadoop/tools/lib/hadoop-streaming-*.jar \
  -D mapreduce.job.reduces=2 \
  -D mapreduce.input.fileinputformat.split.maxsize=524288 \
  -input /user/jurilee/input/DDIA.txt \
  -output /user/jurilee/output \
  -mapper /scripts/mapper.py \
  -reducer /scripts/reducer.py \
  -combiner /scripts/reducer.py \
  -file /scripts/mapper.py \
  -file /scripts/reducer.py
```

각 부분의 의미는 다음과 같다.

- `hadoop jar <jar파일>` 형태로 Hadoop 위에서 Java 프로그램(jar)을 실행하는 명령이다. 
  - `hadoop-streaming-*.jar`는 Hadoop이 기본 제공하는 유틸리티로, Java로 Mapper/Reducer 클래스를 직접 안 짜고 아무 실행 파일(여기선 Python 스크립트)이나 mapper/reducer로 꽂아 쓸 수 있게 해주는 역할을 한다.
  - `$HADOOP_HOME`은 Hadoop 설치 경로 환경변수, 
  - `*`는 버전에 따라 파일명이 달라서(`hadoop-streaming-3.3.6.jar` 등) 와일드카드로 매칭한 것이다.

- `-D key=value`는 Hadoop 설정값을 이 job에만 임시로 지정하는 방식이다. `mapreduce.job.reduces=2`는 reduce task를 몇 개 띄울지 지정하는 것으로, 2로 설정해서 worker 2대가 각각 reduce 작업을 나눠 맡도록 유도한다. (기본값이 낮으면 reducer가 1개만 떠서 병렬화가 안 됨)

- `mapreduce.input.fileinputformat.split.maxsize=524288`은 입력 파일을 몇 바이트 단위로 쪼갤지(split) 지정하는 것이다. 524288바이트 = 512KB. 이 값이 없으면 Hadoop이 기본 블록 크기(보통 128MB) 단위로 쪼개려는데, DDIA.txt가 그보다 작으면 통째로 split 1개 → mapper 1개만 뜬다. 이 옵션으로 강제로 잘게 쪼개서 mapper를 여러 개 병렬로 띄우기 위한 설정이다.

- `-input /user/jurilee/input/DDIA.txt`는 HDFS 상의 입력 경로다. Hadoop이 이 파일을 읽어서 위 split 설정대로 조각내 각 mapper에게 나눠준다.

- `-output /user/jurilee/output`은 결과를 저장할 HDFS 경로다. 주의할 점: 이 디렉토리가 이미 존재하면 Hadoop이 실행을 거부한다(기존 결과를 실수로 덮어쓰지 않기 위한 안전장치). 재실행할 땐 `hdfs dfs -rm -r /user/jurilee/output`로 먼저 지워야 한다.

- `-mapper /scripts/mapper.py`는 Mapper 역할을 할 실행 파일을 지정하는 것이다. Hadoop이 입력 데이터를 한 줄씩 이 프로그램의 stdin으로 넣어주고, 이 프로그램이 stdout으로 내놓는 `단어\t1` 라인들을 mapper 출력으로 받아간다.

- `-reducer /scripts/reducer.py`는 Reducer 역할을 할 실행 파일이다. 정렬된 `단어\t1` 라인들을 stdin으로 받아서, stdout으로 `단어\t총개수`를 출력하는 그 스크립트다.

- `-combiner /scripts/reducer.py`의 Combiner는 각 mapper가 실행되는 노드에서, 데이터를 reducer로 넘기기(shuffle) 전에 미리 한 번 부분 집계를 하는 "미니 reducer"다. word count는 부분합을 나중에 또 더해도 결과가 똑같기 때문에(결합법칙), reducer.py를 그대로 combiner로 재사용할 수 있다. 이걸 넣으면 네트워크로 전송되는 중간 데이터량이 줄어서 더 효율적이다.

- `-file /scripts/mapper.py`, `-file /scripts/reducer.py`는 이 스크립트 파일들을 job과 함께 클러스터 전체에 배포하라는 옵션이다. worker1, worker2는 `/scripts/mapper.py`라는 경로를 원래 갖고 있지 않으니(master 컨테이너에만 있는 파일), `-file`로 지정해줘야 Hadoop이 이 파일을 각 worker 노드로 복사해서 실행할 수 있게 해준다. 이거 빠뜨리면 worker에서 "파일을 찾을 수 없다"는 에러가 난다.

- 각 줄 끝의 `\`는 셸에서 명령어를 여러 줄로 나눠 쓰기 위한 줄바꿈 이스케이프일 뿐, 실제로는 한 줄짜리 명령이다.

<br>

## 6. 분산 처리가 실제로 됐다는 증거를 확보하는 법

job 실행 후 카운터(`Launched map tasks=2`, `Launched reduce tasks=2`, `Combine input/output records`)만으로도 병렬 실행과 combiner 동작을 1차 확인할 수 있지만, **어느 물리 노드(worker1/worker2)에서 돌았는지**까지 보려면:

- YARN 웹 UI(`:8088`) → 해당 Application 클릭 → Application Attempt 화면에서 AM이 실행된 `Node`, `Node Local`/`Off Switch` 컨테이너 수 확인
- job이 **완료된 뒤**에는 `yarn container -list <appattempt_id>`가 항상 빈 목록(`Total number of containers: 0`)을 반환한다 — 이건 실시간 조회용 API라 끝난 job엔 못 쓰는 것이지 분산이 안 됐다는 뜻이 아니다
- 완료된 job의 컨테이너별 실행 노드를 확인하려면 집계 로그를 봐야 한다:
  ```bash
  yarn logs -applicationId <application_id>
  ```
  각 컨테이너 로그 섹션 헤더에 실행 노드(`worker1_...`, `worker2_...`)가 그대로 찍혀 나온다.

<br>

## 7. 결과 검증 — 로컬 파이프라인과 HDFS 결과를 대조

Hadoop 클러스터에 제출하기 전에 반드시 **로컬에서 파이프로 먼저 검증**한다.

```bash
cat DDIA.txt | ./mapper.py | sort | ./reducer.py > local_result.txt
```

그 다음 HDFS 최종 결과(`part-00000`, `part-00001`을 합친 것)와 **고유 단어 수를 비교**해서 완전히 일치하는지 확인했다 (둘 다 11704). 이 두 값이 같다는 건 로직이 맞다는 강력한 근거가 된다. reducer 수를 2로 지정했으므로 출력 파일도 `part-00000`/`part-00001` 두 개로 나뉘어 생성되는데, 이것 자체가 "reducer 2개가 병렬로 결과를 나눠 썼다"는 증거이기도 하다.

# 트러블슈팅

## Python 스크립트 흔한 버그 3종

- **`return` 누락**: 값을 계산만 하고 반환을 안 하면 함수가 암묵적으로 `None`을 리턴한다. `for word in self.tokenize(line):`처럼 그 결과를 순회하려 하면 `TypeError: 'NoneType' object is not iterable`.
- **`\t`와 `/t` 혼동**: `f"{word}/t1"`은 탭이 아니라 문자 그대로 `/t1`을 출력한다. reducer가 `split('\t', 1)`로 파싱하는데 애초에 탭이 없으면 파싱이 깨진다. 반드시 백슬래시 이스케이프 `\t`를 써야 한다.
- **문자열 vs 정수**: `split('\t', 1)`의 결과는 문자열 리스트다. `count`를 `int()`로 변환하지 않고 `current_count += count`를 하면, 초기값이 정수 `0`이라 `TypeError: unsupported operand type(s) for +=: 'int' and 'str'`가 난다.

<br>

## `BrokenPipeError`는 버그가 아니다

```bash
... | python3 reducer.py | head -20
```
`head`가 20줄만 읽고 자기 stdin 파이프를 닫아버리는데, reducer는 그것도 모르고 계속 출력하려다 `SIGPIPE`를 받아서 `BrokenPipeError`가 뜬다. 파이프라인 로직 문제가 아니라 `head`로 자를 때 항상 나타나는 흔한 현상. 결과를 파일로 리다이렉트한 뒤 `head`로 보면 안 나타난다.

<br>

## 유니코드 스마트 따옴표 때문에 단어가 쪼개짐

정식 출판된 전자책(PDF에서 추출한 텍스트)은 `'`(직선 아포스트로피)가 아니라 `’`(U+2019, 스마트 따옴표)를 쓰는 경우가 많다. 토큰화 정규식이 `[\w']+`처럼 직선 아포스트로피만 포함하면 `don’t` 같은 단어가 `don`과 `t`로 쪼개진다. 

`[\w'’]+`처럼 스마트 따옴표도 문자 클래스에 포함시켜야 한다.
