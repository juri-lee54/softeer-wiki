## hadoop user

`ENV HDFS_NAMENODE_USER=root`

- user 많아지면 권한 문제가 생기기 쉬움

## W3M4

### Sentiment Classification

| **Method** | **Pros** | **Cons**  | **Remarks** |
| --- | --- | --- | --- |
| Pre-defined Keyword  | 구현이 매우 간단하고 빠름 | 문맥, 신조어, 비꼬는 표현
등을 전혀 이해하지 못함. |  |
| TF-IDF + KMeans  | 자동 분류, 키워드 불필요  | 클러스터 해석 어려움  | 단어 중요도를 수치화하고 클러스터링 |
| LDA 토픽 모델링  | 의미 기반 분류 가능  | 파라미터 튜닝 필요  | 문서에서 추출된 주제 기반 으로 그룹화 |
| Word2Vec/FastText  | 의미적 유사성 파악  | 임베딩 품질 의존적  | 단어 임베딩으로 유사도 기반 분류 |
| Rule-based 방식
(TextBlob)  | 해석 용이, 설명력 좋음  | 복잡한 패턴에서 어려움  | 패턴 또는 정규식 기반 규칙 분류 |
| 머신러닝 분류기  | 높은 정확도 가능  | 라벨링 데이터 필요  | SVM, RandomForest,
Naive Bayes 등 지도학습 |
- 이 데이터에는 이러한 일반론적 정리들이 안 맞을 수 있음
- 데이터를 샘플링해서 다 돌려보고 가장 좋은 것을 선택해야 함
    - 데이터 특성을 잘 이해해서 대표성을 가지는 데이터를 잘 샘플링 해야 함
        - 샘플링을 잘 못 하면 전체 데이터를 사용했을 때 예상과 다른 결과가 나옴

### Where to Perform Classification

- inside Mapper
- Pre-processing before MapReduce
- External API Call in Mapper
- Spark + MLlib

| **Method** | **Pros** | **Cons**  | **Remarks** |
| --- | --- | --- | --- |
| Inside Mapper  | 분산 처리를 그대로 활용 가능
실시간 분류 가능 | 모든 노드에 모델 환경 구성 필요
배포 및 디버깅 복잡 | Hadoop Streaming, Python 기반 모델 연동 가능 |
| Pre-processing
before MapReduce | 모델 환경 독립적
설정 간단하고 운영 안정적 | Hadoop의 분산처리 장점 일부 상실
사전 처리 시간이 오래 걸릴 수 있음 | 미리 감정 분류한 결과를 HDFS에 저장 후 분석 |
| External API Call in Mapper | 중앙 집중형 모델 관리 용이
모델 업데이트가 쉬움 | 네트워크 I/O 병목
API latency, 실패 위험 존재 | REST API 또는 모델 서버 필요
대규모 분산에 부적합 |
| Spark + MLlib  | 분산 처리와 ML 통합 가능
ML pipeline 구축 쉬움 | Hadoop보다 인프라 설정 복잡
Spark 클러스터 필요 | PySpark UDF 등으로 감정 분석 모델 적용 가능 |
- Spark가 사기템이긴 함
- 프로토타이핑의 목적이 데이터 정합성 확인이면
    - 샘플링 데이터로 프로토타이핑할 때의 방법을 전체 데이터를 사용할 때의 방법과 동일하게 할 필요는 없음
- 프로토타이핑의 목적이 아키텍처의 테스트이면
    - 방법을 동일하게 해서 개선
- 그때 그때 좋은 방법을 선택 == 유연해져야 한다

### Good Business Problem

- We seek a problem with **high monetary value** for **specific people** in certain **situation (context)** who should be willing to pay due to the severe pain that
the problem imposes.
- Basically, we focus on a **small segment (specific people)**, and it should be expandable to a bigger market (more segments) later

<br>

### Good Data Engineering Problem

- For this course
    - should **use Big Data**. (maybe not for the 1st segment but for the whole market)
    - should get **insights from Data**
        - AI might help but not mandatory
    - might end up with **big data processing** (eg. web scraping + data processing)
- 어쨌든 파이프라인을 잘 만드는 것에 대한 평가
    - 이 파이프라인을 설계한 이유?

# 모르는 용어

### POI(관심 지점, Point of Interest)

: 지도나 내비게이션 시스템에서 식당, 병원, 은행, 관광 명소 등 특정 위치나 주요 시설물을 나타내는 데이터 포인트

- **지도 및 내비게이션:** 목적지 설정, 주변 편의시설 검색 등 위치기반 서비스(LBS)의 기본 단위.
- **공간 데이터 분석:** 공공 및 민간에서 상권 분석, 인프라 평가 등의 기반 데이터로 활용.

<br>

### CQRS(Command and Query Responsibility Segregation, 명령-쿼리 책임 분리)

: 애플리케이션의 데이터 **변경 작업**(쓰기/수정/삭제)과 **조회 작업**(읽기)을 서로 다른 모델과 저장소로 분리하여, 각각 독립적으로 최적화하고 확장할 수 있게 해주는 아키텍처 패턴

- **핵심 개념**
    - **명령(Command):** 시스템의 상태를 변경하는 작업(CUD)
        - 비즈니스 로직과 데이터의 유효성 검증에 집중하며, 보통 값을 반환하지 않거나 성공 여부만 반환
    - **쿼리(Query):** 시스템의 상태를 변경하지 않고 오직 데이터를 읽어오는 작업
        - 화면에 필요한 데이터(DTO)를 빠르고 효율적으로 가져오는 데 집중
- 왜 사용?
    - 기존의 CRUD 방식은 읽기와 쓰기가 같은 모델과 데이터베이스를 공유하여 트랜잭션 경합과 성능 저하가 발생하기 쉬움
    - **성능 및 확장성:** 조회 트래픽이 몰리는 서비스(예: 상품 목록)와 쓰기가 많은 서비스(예: 주문 처리)를 독립적으로 확장
    - **최적화된 스키마:** 쓰기 모델은 복잡한 비즈니스 규칙과 데이터 무결성에 맞추고, 읽기 모델은 UI에 최적화된 구조(예: NoSQL 활용)로 분리
- **단순 분리(Simple/Premium):** 애플리케이션 레벨에서 로직과 데이터베이스 테이블을 분리하는 방식
- **이벤트 소싱(Event Sourcing):** 쓰기 모델에서 발생한 모든 변경 이벤트를 발행하고, 이벤트를 구독하여 읽기 전용 저장소(Read Store)를 업데이트하는 방식
