# 미션 설명

Docker로 Apache Spark **Standalone 클러스터**(마스터 1대 + 워커 2대)를 직접 구축하고, 몬테카를로 방식으로 원주율(π)을 추정하는 Spark 잡을 그 클러스터에 제출해보는 미션. **"Spark 클러스터라는 실행 환경 자체를 어떻게 코드로 재현 가능하게 만드는가"** 를 다루는 인프라 성격의 미션이다.

핵심 파일 4개와 그 역할:

| 파일 | 역할 |
|---|---|
| `Dockerfile` | 마스터/워커가 공유하는 이미지를 정의 (Java + Spark 설치) |
| `entrypoint.sh` | 컨테이너가 뜰 때 마스터로 동작할지 워커로 동작할지 분기 |
| `docker-compose.yml` | 컨테이너 3개(마스터 1, 워커 2)를 하나의 네트워크로 묶어 클러스터 구성 |
| `submit.sh` | 마스터 컨테이너 안에서 `spark-submit`을 실행해 잡을 제출 |

<br>

# 전체 구조 한눈에 보기

```
호스트(내 컴퓨터)
  │  docker-compose up --build -d
  ▼
spark-net (Docker bridge 네트워크)
  │
  ├─ spark-master   (컨테이너)  ── 웹 UI :8080, 통신 포트 :7077
  ├─ spark-worker-1 (컨테이너)  ── 웹 UI :8081
  └─ spark-worker-2 (컨테이너)  ── 웹 UI :8082(호스트) → 8081(컨테이너 내부)

세 컨테이너 모두 같은 이미지(Dockerfile)에서 빌드되고,
SPARK_MODE 환경변수 값(master/worker)만 다르게 줘서 역할을 나눈다.
```

컨테이너 3개는 전부 **동일한 이미지**에서 시작한다. "마스터 이미지 따로, 워커 이미지 따로"가 아니라, **환경변수 하나(`SPARK_MODE`)로 같은 이미지가 다른 역할을 하도록** 만든 게 이 구조의 핵심이다.

<br>

# 1. Dockerfile: 이미지 하나로 마스터/워커를 겸용하기

## 베이스 이미지: `eclipse-temurin:17-jdk-jammy`

```dockerfile
FROM eclipse-temurin:17-jdk-jammy
```

Spark는 JVM 위에서 동작하는 프로그램이라 Java(JDK)가 필요하다. `openjdk` 공식 이미지는 Docker Hub에서 더 이상 유지보수되지 않아(deprecated) 대신 이클립스 재단이 관리하는 `eclipse-temurin`을 쓴다. 버전은 17을 선택했는데, **Spark 4.x부터 Java 11 지원이 종료**되어 최소 Java 17 이상이 필요하기 때문이다.

`eclipse-temurin` 이미지는 `JAVA_HOME`(`/opt/java/openjdk`)을 이미지 자체에서 기본 제공한다. 그래서 예전 `openjdk:11-slim` 베이스를 쓸 때 필요했던 "JDK 심볼릭 링크 직접 생성" 같은 작업이 필요 없다.

## Spark 배포판 직접 다운로드

```dockerfile
ARG SPARK_VERSION=4.2.0
ARG HADOOP_VERSION=3

RUN wget https://dlcdn.apache.org/spark/spark-${SPARK_VERSION}/spark-${SPARK_VERSION}-bin-hadoop${HADOOP_VERSION}.tgz -O /tmp/spark.tar.gz \
    && tar -xzf /tmp/spark.tar.gz -C /opt/ \
    && mv /opt/spark-${SPARK_VERSION}-bin-hadoop${HADOOP_VERSION} ${SPARK_HOME} \
    && rm /tmp/spark.tar.gz
```

Spark를 패키지 매니저(apt 등)로 설치하는 대신, **공식 배포 tarball을 직접 받아 압축을 푸는 방식**을 쓴다. 이렇게 하면 `ARG SPARK_VERSION`/`ARG HADOOP_VERSION`으로 정확히 원하는 버전을 고정할 수 있고, `docker build --build-arg SPARK_VERSION=...`로 나중에 버전만 바꿔 재빌드하기도 쉽다.

다운로드 주소는 `archive.apache.org`(과거 버전 보관용, 느림) 대신 CDN이 붙은 `dlcdn.apache.org`(공식 배포 서버)를 쓴다.

## 환경변수로 Spark 커맨드를 어디서든 쓸 수 있게 함

```dockerfile
ENV SPARK_HOME=/opt/spark
ENV PATH=$SPARK_HOME/bin:$SPARK_HOME/sbin:$PATH
```

`SPARK_HOME`을 지정하고, `bin/`(spark-submit, spark-shell 등)과 `sbin/`(start-master.sh 등)을 `PATH`에 추가해서 컨테이너 안 어느 디렉터리에서든 `spark-submit`, `spark-class` 같은 명령을 바로 쓸 수 있게 한다.

## ENTRYPOINT로 진입점을 스크립트에 위임

```dockerfile
COPY entrypoint.sh /opt/entrypoint.sh
RUN chmod +x /opt/entrypoint.sh

EXPOSE 7077 8080 8081

ENTRYPOINT ["/opt/entrypoint.sh"]
```

이미지 자체는 "마스터가 될지 워커가 될지" 모른다. 그 결정은 컨테이너가 실제로 뜰 때(`docker run`/`docker-compose up` 시점)에 전달되는 환경변수로 정해지고, 그 분기 로직은 `entrypoint.sh`에 있다. `EXPOSE`는 포트를 실제로 여는 게 아니라 "이 컨테이너가 이 포트들을 쓴다"는 문서화 역할만 하고, 실제 호스트 포트 매핑은 `docker-compose.yml`에서 한다.

# 2. entrypoint.sh: 컨테이너 하나가 master/worker 중 뭐가 될지 정하는 스위치

```bash
#!/bin/bash
set -e

if [ "$SPARK_MODE" = "master" ]; then
    exec /opt/spark/bin/spark-class org.apache.spark.deploy.master.Master \
        --host 0.0.0.0 \
        --port 7077 \
        --webui-port 8080

elif [ "$SPARK_MODE" = "worker" ]; then
    exec /opt/spark/bin/spark-class org.apache.spark.deploy.worker.Worker \
        "$SPARK_MASTER_URL" \
        --webui-port 8081

else
    echo "SPARK_MODE 환경변수를 master 또는 worker 로 지정해주세요. (현재 값: '$SPARK_MODE')"
    exit 1
fi
```

## `SPARK_MODE` 환경변수로 분기

`docker-compose.yml`이 컨테이너별로 `SPARK_MODE=master` 또는 `SPARK_MODE=worker`를 넘겨주고, 이 스크립트가 그 값을 보고 마스터 데몬을 띄울지 워커 데몬을 띄울지 결정한다. 값이 `master`/`worker`가 아니면 즉시 에러 메시지를 출력하고 `exit 1`로 종료한다 — 잘못된 설정으로 컨테이너가 알 수 없는 상태로 떠 있는 것보다, 바로 실패하는 쪽이 디버깅하기 쉽다.

## 왜 `start-master.sh` 대신 `spark-class`를 직접, `exec`로 실행하는가

Spark 배포판에는 `sbin/start-master.sh`, `sbin/start-worker.sh`처럼 데몬을 백그라운드로 띄워주는 편의 스크립트가 이미 있다. 그런데 이 스크립트들은 프로세스를 **백그라운드**로 fork해서 띄운 뒤 즉시 종료(return)하도록 설계되어 있다 — 일반 서버(VM)에서는 "데몬을 띄우고 터미널은 돌려받는다"는 게 자연스럽지만, **Docker 컨테이너는 PID 1 프로세스가 종료되는 순간 컨테이너 자체가 종료된다.** `start-master.sh`를 그대로 쓰면 스크립트가 데몬을 백그라운드로 띄우자마자 자기 자신은 끝나버리고, 컨테이너도 같이 죽어버린다.

그래서 이 미션은 편의 스크립트를 안 쓰고, 데몬의 실제 진입점 클래스(`org.apache.spark.deploy.master.Master` / `org.apache.spark.deploy.worker.Worker`)를 `spark-class`로 **포그라운드에서 직접** 실행한다. 여기에 `exec`까지 붙인 이유는, `exec` 없이 그냥 실행하면 `spark-class` 프로세스가 `entrypoint.sh`(bash)의 **자식 프로세스**로 남아서 PID 1은 여전히 bash가 되고, `docker stop`이 보내는 종료 시그널이 자식 프로세스까지 제대로 전달 안 될 수 있다. `exec`는 현재 셸 프로세스를 그 커맨드로 **치환**해서, `spark-class`(JVM)가 그대로 PID 1이 되게 만든다. 그래야 컨테이너 시그널 처리와 생명주기가 예상대로 동작한다.

# 3. docker-compose.yml: 컨테이너 3개를 하나의 클러스터로 묶기

## 같은 이미지, 다른 환경변수로 역할만 분리

```yaml
spark-master:
  build: { context: ., dockerfile: Dockerfile }
  environment:
    - SPARK_MODE=master

spark-worker-1:
  build: { context: ., dockerfile: Dockerfile }   # 마스터와 완전히 동일한 이미지
  environment:
    - SPARK_MODE=worker
    - SPARK_MASTER_URL=spark://spark-master:7077
```

세 서비스 모두 `build.dockerfile`이 같다. 다른 건 오직 `environment`뿐이다. 이미지를 하나만 관리하면 되니 빌드/유지보수가 단순해지고, "마스터 이미지와 워커 이미지의 Spark 버전이 어긋나는" 종류의 문제 자체가 생길 수 없다.

## 네트워크: 서비스명이 곧 호스트명이 되는 이유

```yaml
networks:
  spark-net:
    driver: bridge
```

세 서비스를 같은 `spark-net`(사용자 정의 bridge 네트워크)에 연결해두면, Docker가 제공하는 내장 DNS 덕분에 **서비스명 자체가 호스트명으로 resolve된다.** 그래서 워커 설정의 `SPARK_MASTER_URL=spark://spark-master:7077`처럼, 마스터 컨테이너의 실제 IP를 몰라도 서비스명(`spark-master`)만으로 접속할 수 있다. 컨테이너를 재시작해서 내부 IP가 바뀌어도 서비스명은 그대로라 설정을 바꿀 필요가 없다.

## 볼륨 마운트 3종(jobs/data/output)

```yaml
volumes:
  - ./jobs:/opt/jobs      # spark-submit으로 넘길 작업 스크립트
  - ./data:/opt/data      # 작업 입력 데이터셋
  - ./output:/opt/output  # 작업 결과 (호스트에서 바로 확인 가능)
```

이 세 볼륨을 **마스터/워커 컨테이너 전부에 동일하게 마운트**한 게 중요하다. Standalone 클러스터에서 각 워커는 독립된 프로세스이기 때문에, 워커가 잡 코드(`/opt/jobs/pi.py`)나 입력 데이터(`/opt/data`)에 접근하려면 그 워커 컨테이너 자신도 같은 경로에 파일이 있어야 한다. 호스트 폴더 하나를 여러 컨테이너에 동시에 마운트해두면, 호스트 입장에선 파일 하나만 관리해도 모든 컨테이너에서 동일한 내용을 보게 된다. `output`도 마찬가지로, 어느 워커가 그 파티션을 처리해서 결과를 쓰든 호스트의 `./output`에 바로 나타난다.

## 포트 매핑: 워커 2대가 같은 내부 포트(8081)를 쓰는데 충돌 안 나는 이유

```yaml
spark-worker-1:
  ports:
    - "8081:8081"
spark-worker-2:
  ports:
    - "8082:8081"   # 호스트 8082 → 컨테이너 내부는 똑같이 8081
```

`entrypoint.sh`는 워커를 띄울 때 항상 `--webui-port 8081`로 고정해서 실행한다. 즉 **컨테이너 내부에서는** worker-1과 worker-2 둘 다 8081번 포트를 쓴다. 컨테이너는 각자 독립된 네트워크 네임스페이스를 가지므로 이건 문제가 안 되지만, 그 포트를 호스트로 매핑할 때는 호스트 쪽 포트가 겹치면 안 된다. 그래서 `"호스트포트:컨테이너포트"` 형식에서 컨테이너 쪽은 둘 다 8081로 두고, 호스트 쪽만 8081/8082로 다르게 잡아 충돌을 피한다.

## `depends_on`이 보장하는 것과 안 하는 것

```yaml
spark-worker-1:
  depends_on:
    - spark-master
```

`depends_on`은 **컨테이너가 시작되는 순서**만 보장한다 (마스터 컨테이너를 먼저 만들고 워커 컨테이너를 그 다음에 만듦). 마스터 컨테이너 "안의 Spark Master 프로세스가 7077 포트에 바인딩을 끝냈는지"까지는 보장하지 않는다. 다만 Spark 워커는 마스터 연결에 실패하면 자동으로 재시도하도록 만들어져 있어서, 이 미션에서는 별도의 헬스체크 없이도 실질적으로 문제없이 동작한다. (워커 로그에 연결 재시도 메시지가 반복된다면, 이 재시도가 실패하고 있다는 뜻이므로 `SPARK_MASTER_URL` 값이나 마스터 컨테이너 상태부터 확인해야 한다.)

# 5. spark-submit으로 작업 제출하기

```bash
docker exec spark-master \
    spark-submit \
    --master spark://spark-master:7077 \
    --deploy-mode client \
    /opt/jobs/pi.py "${INPUT_PATH}" "${OUTPUT_PATH}"
```

## 왜 호스트에서 바로 `spark-submit`을 못 하는가

Spark는 이미지 **내부**(`/opt/spark`)에만 설치돼 있고, 호스트 컴퓨터에는 Spark가 전혀 설치돼 있지 않다. `spark-submit`이라는 명령 자체가 호스트 PATH에 없으니, `docker exec <컨테이너 이름> spark-submit ...`으로 **이미 Spark가 설치된 컨테이너 안에서** 명령을 대신 실행시켜야 한다. `spark-master` 컨테이너를 고른 이유는 어차피 클러스터의 일부라 접속 정보(`spark://spark-master:7077`)를 알고 있고, `jobs`/`data`/`output` 볼륨도 이미 마운트돼 있기 때문이다.

## `--master` / `--deploy-mode client`의 의미

- `--master spark://spark-master:7077`: 이 잡을 어떤 클러스터 매니저에게 제출할지 지정. Standalone 클러스터의 마스터 주소.
- `--deploy-mode client`: **드라이버 프로세스가 어디서 도는가**를 결정. `client` 모드는 `spark-submit`을 실행한 그 프로세스(여기선 `spark-master` 컨테이너 안) 자체가 드라이버가 되고, 실제 연산(익스큐터)은 워커 컨테이너들에서 돈다. (반대인 `cluster` 모드는 드라이버 자체도 클러스터의 워커 노드 중 하나에서 실행된다 — 이 미션은 로컬 스크립트 결과를 바로 로그로 보고 싶어서 `client`를 씀.)

<br>

# 6. 실습 잡(pi.py)에 담긴 RDD 개념

## `parallelize(range(n), partitions)`: 데이터를 파티션 단위로 쪼개기

```python
n = samples_per_partition * partitions
partition_results = spark.sparkContext.parallelize(range(n), partitions) \
    .mapPartitionsWithIndex(count_hits_in_partition)
```

`range(n)`이라는 숫자 시퀀스를 `partitions`개의 파티션으로 나눠서 RDD를 만든다. 파티션 수를 워커 수(2)에 맞추면, 각 파티션이 워커 한 대씩에 배정돼 병렬로 처리된다. 파티션 수가 실제 워커 수보다 많으면 한 워커가 여러 파티션을 순차 처리하고, 데이터 양보다 파티션 수가 많으면 일부 파티션이 비어서 빈 결과 파일이 생길 수 있다.

## `mapPartitionsWithIndex`: 원소 단위가 아니라 파티션 단위로 처리하기

일반적인 `map()`은 원소 하나하나에 함수를 적용하지만, `mapPartitionsWithIndex(f)`는 **파티션 전체(iterator)와 그 파티션의 인덱스**를 함수에 넘긴다. 그래서 파티션 안에서 상태(hit 카운트 등)를 누적하며 처리하고, 파티션당 결과를 한 번만(`yield`) 내보낼 수 있다 — 매 원소마다 함수 호출 오버헤드가 생기는 `map()`보다 이런 "파티션 단위 집계" 작업에 더 적합하다.

## 파티션마다 랜덤 시드를 따로 주는 이유 (fork와 난수 상태 상속)

```python
def count_hits_in_partition(partition_id, iterator):
    # PySpark 워커 프로세스는 fork로 뜨기 때문에 random 모듈이 부모와 같은 시드 상태를 물려받음
    # partition_id로 명시적으로 시드를 다르게 줘서 파티션마다 다른 난수 시퀀스가 나오게 함
    rng = random.Random(partition_id)
```

PySpark의 워커 프로세스는 부모 프로세스를 `fork`해서 만들어지는데, 이때 파이썬 `random` 모듈의 내부 시드 상태도 그대로 복제된다. 만약 각 파티션에서 그냥 전역 `random.random()`을 썼다면, 여러 파티션이 **동일한 난수 시퀀스**를 만들어낼 위험이 있다 — 몬테카를로 시뮬레이션에서 이건 치명적이다(파티션마다 진짜 독립적인 샘플이어야 결과가 의미 있음). 그래서 `random.Random(partition_id)`로 파티션마다 명시적으로 다른 시드를 주는 별도의 RNG 인스턴스를 만들어 이 문제를 피한다.

## 출력 파일 개수 = 파티션 개수

```python
result_df.write.mode("overwrite").option("header", True).csv(output_path)
```

`coalesce()`나 `repartition()` 없이 그대로 저장하면, DataFrame의 파티션 개수만큼 출력 파일(`part-00000`, `part-00001`, ...)이 생긴다. 이 미션은 파티션 수를 워커 수(2)에 맞춰뒀기 때문에, 결과 폴더에 정확히 2개의 CSV가 남는다.

# 실무 팁 / 트러블슈팅

- **미션마다 컨테이너 이름/포트가 고정돼 있다.** `container_name: spark-master/spark-worker-1/spark-worker-2`와 포트(`7077`/`8080`/`8081`/`8082`)가 하드코딩돼 있어서, 다른 미션(W4M2 등) 클러스터가 이미 떠 있는 상태에서 이 미션을 `up`하면 컨테이너 이름/포트 충돌로 실패한다. 새 미션 클러스터를 올리기 전에 `docker ps`로 다른 미션 컨테이너가 떠 있는지 먼저 확인하고 `docker-compose down`으로 내려야 한다.
- **워커가 마스터에 안 붙을 때**: `docker logs spark-worker-1`에 연결 재시도 메시지가 반복되면 `SPARK_MASTER_URL` 값(서비스명 오타, 포트 번호)과 마스터 컨테이너가 실제로 `Up` 상태인지부터 확인한다.
- **`entrypoint.sh`가 에러 메시지만 찍고 바로 죽어있다면**: `SPARK_MODE`가 `master`/`worker` 둘 중 하나로 정확히 설정됐는지 `docker-compose.yml`의 `environment`를 확인한다 (오타에 특히 취약한 지점).
- **파티션 수와 워커 수를 맞추는 습관**: `data/pi_config.csv`의 `partitions` 값을 워커 수와 다르게 주면, 워커 하나가 노는 상태로 클러스터가 덜 활용되거나(파티션 < 워커 수), 한 워커가 여러 파티션을 순차 처리하게 된다(파티션 > 워커 수).
