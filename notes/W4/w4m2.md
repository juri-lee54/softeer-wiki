# 미션 설명

뉴욕시 TLC(Taxi & Limousine Commission) 운행 기록을 Spark로 수집·정제·분석하고 Jupyter Notebook으로 시각화하는 미션. 클러스터(마스터 1 + 워커 2) 자체는 W4M1에서 만든 Docker Standalone 구성을 그대로 재사용했고, 이번에 새로 한 일은 그 위에서 돌아가는 **실제 DataFrame 기반 데이터 파이프라인**이다.

만든 job은 순서대로 4개:
1. `clean_trips.py` — TLC 운행 기록 로딩 + 결측값/이상치 클리닝
2. `compute_metrics.py` — 평균 이동시간/거리 집계
3. `peak_hours.py` — 시간대별 운행 분포 + 피크 시간대 계산
4. `weather_correlation.py` — 기상 데이터와 운행 수요 상관관계 통계 검증

# Spark DataFrame API 핵심 개념

W4M1의 `pi.py`는 `sparkContext.parallelize()` + `mapPartitionsWithIndex()`로 저수준 RDD API만 썼다. 이번 미션은 4개 job을 전부 실무에서 훨씬 많이 쓰는 **Spark SQL DataFrame API**(`spark.read`, `filter`, `withColumn`, `groupBy().agg()`, `join`, `write`)로 작성했는데, DE-20260720.md는 이 API 자체가 아니라 Spark의 아키텍처(Driver/Executor, 클러스터 매니저)를 다뤘어서, DataFrame API 고유의 개념은 따로 정리한다.

## Transformation과 Action, 그리고 지연 평가(Lazy Evaluation)

`filter`/`withColumn`/`groupBy`/`join`처럼 새 DataFrame을 반환하는 연산(Transformation)은 그 자리에서 실행되지 않고 실행 계획(DAG)만 쌓인다. `count()`/`collect()`/`write()`/`first()`처럼 실제 값을 만들어내는 연산(Action)을 호출해야 그 시점에 계획 전체가 실행된다.

이걸 놓쳤을 때 생기는 문제를 `clean_trips.py`에서 실제로 만들었다.

```python
df = spark.read.parquet(input_path)
raw_count = df.count()              # action 1 - parquet 전체를 한 번 스캔

cleaned_df = df.filter(...)
cleaned_count = cleaned_df.count()  # action 2 - 캐시가 없어서 처음부터 다시 읽고 필터링
...
cleaned_df.write...(output_path)    # action 3 - 또 처음부터 다시 읽고 필터링
```

`cleaned_df`를 캐시해두지 않았기 때문에, `count()`와 `write()`를 부를 때마다 **parquet 읽기부터 필터링까지 매번 처음부터 다시 실행된다.** 6개월치(2천만 행) 기준으로 사실상 같은 스캔을 3번 한 셈. 반대로 `peak_hours.py`/`weather_correlation.py`는 `count()`/`collect()`/`write()`/`toPandas()`처럼 액션을 여러 번 부르기 전에 `.cache()`를 걸어 한 번 계산한 결과를 재사용하도록 했다. **같은 DataFrame에 액션을 두 번 이상 부를 계획이면, 그 전에 캐시가 필요한지부터 판단해야 한다.**

## Catalyst 옵티마이저의 Predicate/Column Pushdown

`clean_trips.py` 실행 로그에 이런 줄이 찍혔다.

```
Pushed Filters: IsNotNull(trip_distance),IsNotNull(tpep_pickup_datetime),...,
                GreaterThan(trip_distance,0.0),LessThanOrEqual(trip_distance,100.0)
```

`filter()`로 건 조건을 Spark가 자동으로 Parquet 리더 자체에 밀어넣어서, 필요 없는 행은 디스크에서 읽는 단계부터 걸러낸다(Predicate Pushdown). Parquet은 컬럼 단위로 저장되는 포맷이라 쓰지 않는 컬럼은 아예 읽지도 않는다(Column Pruning). **이 최적화는 코드에서 따로 호출하는 게 아니라, DataFrame API로 짜기만 하면 Catalyst 옵티마이저가 실행 계획을 세울 때 자동으로 적용해준다** — `mapPartitionsWithIndex` 같은 저수준 RDD API에는 없는, DataFrame API 특유의 이점이다.

## CSV는 기본적으로 전부 String으로 읽힌다

`weather_correlation.py`에서 `spark.read.option("header", True).csv(weather_path)`로 기상 데이터를 읽었는데, `inferSchema` 옵션을 안 줬기 때문에 `HourlyDryBulbTemperature`, `HourlyPrecipitation` 같은 숫자 컬럼도 전부 `StringType`으로 들어온다. 그래서 바로 산술 연산을 할 수 없고, `rlike()`로 형식을 검증한 뒤 `.cast(DoubleType())`으로 명시적으로 바꿔야 했다.

```python
F.when(F.col("HourlyDryBulbTemperature").rlike(r"^-?\d+$"),
       F.col("HourlyDryBulbTemperature").cast(DoubleType()))
```

Parquet은 파일 자체에 스키마가 박혀 있어서 `spark.read.parquet()`엔 이런 문제가 없었다(TLC 데이터는 처음부터 컬럼 타입이 맞게 들어옴 — [5번 항목](#5-실제-공개-기상-데이터noaa-lcd는-한-파일에-여러-리포트-타입지저분한-값이-섞여-있다)의 지저분한 값 문제와 겹쳐서 더 까다로웠다). **CSV는 스키마가 없는 텍스트 포맷이라, `inferSchema`를 안 쓰면 전부 문자열로 읽힌다는 걸 전제로 캐스팅 계획을 짜야 한다.**

## coalesce(1)은 "결과를 합치는" 게 아니라 "쓰기 전에 파티션을 줄이는" 것

지표 요약처럼 결과가 몇 줄 안 되는 job(`compute_metrics.py`, `peak_hours.py`, `weather_correlation.py`)은 `coalesce(1).write...csv(...)`로 파일 하나만 나오게 했다. 반면 `clean_trips.py`의 정제된 운행 기록(수백만~수천만 행)은 coalesce 없이 그대로 저장해서 여러 워커가 병렬로 나눠 쓰도록 뒀다.

`coalesce(n)`은 `repartition(n)`과 달리 전체 셔플 없이 같은 워커 안에서 파티션 개수만 줄이는 가벼운 연산이지만, 그렇다고 공짜는 아니다. 큰 데이터를 억지로 파티션 1개로 합치면 그 파티션을 처리하는 워커/코어 하나에 부하가 몰리고 쓰기도 사실상 직렬화된다. **"사람이 파일 하나로 바로 열어봐야 하는 작은 요약 결과"에만 `coalesce(1)`을 쓰고, 대용량 결과는 여러 파일로 나눠 쓰는 걸 기본으로 삼아야 한다.**

# 4개 Job 자세히 뜯어보기

## 1. clean_trips.py — 수집 + 클리닝

### 여러 포맷을 자동으로 찾아 합치기

```python
parquet_paths = sorted(glob.glob(f"{input_dir}/yellow_tripdata_*.parquet"))
csv_paths = sorted(glob.glob(f"{input_dir}/yellow_tripdata_*.csv"))

dfs = []
if parquet_paths:
    dfs.append(spark.read.parquet(*parquet_paths))
if csv_paths:
    dfs.append(spark.read.option("header", True).option("inferSchema", True).csv(*csv_paths))

df = dfs[0]
for other in dfs[1:]:
    df = df.unionByName(other)
```

파이썬 표준 라이브러리 `glob`으로 **드라이버(파이썬 프로세스) 쪽에서** 어떤 파일이 있는지 먼저 확인한 뒤, 있는 포맷만 Spark로 읽어서 `unionByName`으로 합친다. `union()`이 아니라 `unionByName()`을 쓴 이유는, parquet와 CSV를 읽을 때 컬럼 순서가 다를 수 있기 때문이다 — `union()`은 위치(순서) 기준으로 합쳐서 순서가 다르면 엉뚱한 컬럼끼리 합쳐지지만, `unionByName()`은 컬럼 **이름**을 보고 맞춰서 합친다. 파일명 패턴을 `yellow_tripdata_*`로 좁힌 이유는, 같은 `data/` 폴더에 있는 날씨 CSV(`weather_*_central_park.csv`) 같은 다른 종류 파일이 섞여 들어오지 않게 하기 위해서다.

### 시간 필드 표준화 + 이동시간 계산

```python
df = df \
    .withColumn("tpep_pickup_datetime", F.to_timestamp("tpep_pickup_datetime")) \
    .withColumn("tpep_dropoff_datetime", F.to_timestamp("tpep_dropoff_datetime"))

df = df.withColumn(
    "trip_duration_minutes",
    (F.unix_timestamp("tpep_dropoff_datetime") - F.unix_timestamp("tpep_pickup_datetime")) / 60.0
)
```

`to_timestamp`로 문자열/다양한 원본 표현을 Spark의 표준 `TimestampType`으로 통일한다(Parquet은 이미 timestamp 타입으로 들어오지만, CSV는 문자열로 들어오므로 이 캐스팅이 실질적으로 CSV 쪽에 필요하다). 이동시간은 `unix_timestamp()`로 두 타임스탬프를 초 단위 정수(Unix epoch)로 바꾼 뒤 빼고 60으로 나눠 분 단위로 만든다 — 타임스탬프끼리 직접 뺄셈이 안 되기 때문에 거치는 표준적인 패턴이다.

### 결측값: 버릴지 채울지는 컬럼의 역할에 따라 다르게 결정

```python
df = df.na.fill({"passenger_count": 1})       # 채움
...
.filter(... & F.col("tpep_pickup_datetime").isNotNull() & F.col("tpep_dropoff_datetime").isNotNull())  # (사실상) 버림
```

같은 "결측값 처리"라도 컬럼마다 다르게 다뤘다. `passenger_count`(승객 수)는 이 미션의 핵심 지표(이동시간/거리)와 무관한 컬럼이라, 값이 없다고 행 전체를 버리면 정작 필요한 이동시간/거리 데이터까지 같이 손실된다 — 그래서 합리적인 기본값(1명)으로 **채운다**(`na.fill`). 반면 픽업/드롭오프 시각처럼 지표 계산에 직접 쓰이는 핵심 컬럼이 비어 있으면 대체할 방법이 없으므로 그 행 자체를 **버린다**(필터). 즉 "결측값 처리 = 무조건 채우기" 또는 "무조건 버리기"가 아니라, **그 컬럼이 최종 지표 계산에 필수적인지**를 기준으로 방식을 나눠야 한다.

### 이상치 필터링 상/하한

```python
MAX_DURATION_MINUTES = 24 * 60
MAX_DISTANCE_MILES = 100

cleaned_df = df.filter(
    (F.col("trip_duration_minutes") > 0) & (F.col("trip_duration_minutes") <= MAX_DURATION_MINUTES) &
    (F.col("trip_distance") > 0) & (F.col("trip_distance") <= MAX_DISTANCE_MILES) & ...
)
```

미션 스펙은 "음수 이동 시간/거리"만 예로 들었지만, 실제 데이터엔 음수는 없고 대신 **비현실적으로 큰 값**(312,722마일짜리 이동거리, 몇 년 전 날짜의 픽업 시각)이 섞여 있었다. 그래서 하한(0 초과)뿐 아니라 상한도 같이 둬서, "말이 안 되는 큰 값"도 같은 필터 한 번에 걸러내도록 설계했다. 상한값 자체는 임의의 숫자가 아니라 도메인 지식에 근거한다(뉴욕 시내 택시가 하루를 넘겨 운행하는 경우는 사실상 없음, 인접 도시까지의 거리도 100마일을 안 넘음).

## 2. compute_metrics.py — 지표 집계

```python
metrics_row = df.agg(
    F.round(F.avg("trip_duration_minutes"), 2).alias("avg_trip_duration_minutes"),
    F.round(F.avg("trip_distance"), 2).alias("avg_trip_distance_miles"),
    F.count(F.lit(1)).alias("trip_count")
).first()
```

`groupBy` 없이 `agg()`만 부르면 **전체 DataFrame을 하나의 그룹으로 보고** 집계한다(그룹별이 아니라 전체 평균/개수 하나씩). 이렇게 하면 `avg`/`count`처럼 서로 다른 집계 함수 여러 개를 **한 번의 액션**(`.first()`)으로 동시에 계산할 수 있다 — `df.filter(...).count()`와 `df.agg(F.avg(...))`를 따로따로 부르면 그 수만큼 별도의 스캔이 도는데(→ [Transformation/Action 절](#transformation과-action-그리고-지연-평가lazy-evaluation) 참고), 필요한 집계를 한 번의 `agg()` 호출에 몰아넣으면 스캔을 한 번으로 줄일 수 있다. `F.round(..., 2)`로 소수점을 미리 정리해두는 것도 "사람이 읽기 쉬운 형식" 요구사항을 코드 단계에서부터 챙긴 것이다.

## 3. peak_hours.py — 피크 시간대

```python
hourly_df = df \
    .withColumn("pickup_hour", F.hour("tpep_pickup_datetime")) \
    .groupBy("pickup_hour") \
    .agg(F.count(F.lit(1)).alias("trip_count")) \
    .orderBy("pickup_hour")
```

`F.hour()`는 타임스탬프에서 **시(0~23)만** 뽑아낸다 — 날짜는 버리고 시각만 남기 때문에, 몇 달치 데이터를 넣어도 "1월 3일 18시"와 "3월 20일 18시"가 같은 그룹(`pickup_hour=18`)으로 합쳐진다. 이게 미션이 정의한 "피크 시간대"(시간당 출발 차량 수)의 핵심이다 — 특정 날짜가 아니라 **하루 24시간 중 어느 시간대**가 가장 붐비는지를 보는 것이므로, 날짜를 버리고 시(hour)로만 묶는 게 맞다. (반대로 [weather_correlation.py](#4-weather_correlationpy--기상-상관관계)는 날짜까지 포함한 `date_trunc("hour", ...)`를 쓰는데, 이건 기상 조건이 "그 순간"의 값이라 날짜까지 맞춰야 하기 때문이다 — 같은 "시간 단위 집계"라도 목적에 따라 자르는 방식이 다르다.)

```python
peak_row = max(rows, key=lambda r: r["trip_count"])
result_df = hourly_df.withColumn("is_peak", F.col("pickup_hour") == peak_row["pickup_hour"])
```

24개 행(최대)만 남은 뒤 `collect()`로 드라이버에 모아 파이썬 `max()`로 최댓값 행을 찾는다 — 데이터 양이 24행뿐이라 Spark 연산(`orderBy` + `limit(1)`) 없이 드라이버에서 처리해도 무리가 없다. 찾아낸 피크 시간을 다시 `is_peak` boolean 컬럼으로 원본 DataFrame에 붙여두면, 결과 CSV 자체에 "어디가 피크인지" 정보가 포함돼서 Jupyter 노트북에서 막대그래프 그릴 때 별도 계산 없이 바로 색을 다르게 칠할 수 있다.

## 4. weather_correlation.py — 기상 상관관계

### NOAA LCD 데이터셋 구조

기상 데이터는 [NOAA Local Climatological Data (LCD)](https://www.ncei.noaa.gov/products/land-based-station/local-climatological-data)를 쓴다. 파일 하나가 관측소 하나(여기선 센트럴파크, WBAN 94728, 파일명은 USAF+WBAN을 합친 11자리 station id `72505394728`)의 **연간** 관측 기록이고, 그 안에 `REPORT_TYPE` 컬럼으로 여러 종류의 리포트가 섞여 있다.

| REPORT_TYPE | 의미 |
|---|---|
| `FM-15` | 정시 시간별 관측 (이 미션에서 쓰는 것) |
| `FM-16` | 특별 관측(SPECI, 기상이 급변할 때 추가로 찍힘) |
| `SOD` | 일별 요약(Summary of Day) |
| `SOM` | 월별 요약(Summary of Month) |

시간 단위 분석이 목적이라 `FM-15`만 필터링해서 쓴다(`F.col("REPORT_TYPE").contains("FM-15")`) — 나머지 리포트 타입 행은 애초에 `HourlyDryBulbTemperature` 같은 컬럼 자체가 비어 있거나 다른 주기로 찍히기 때문에 섞어 쓰면 안 된다.

### 문자열 컬럼 정제 패턴

```python
F.when(F.col("HourlyDryBulbTemperature").rlike(r"^-?\d+$"),
       F.col("HourlyDryBulbTemperature").cast(DoubleType()))

F.when(F.col("HourlyPrecipitation").isin("T", ""), F.lit(0.0))
 .otherwise(F.regexp_replace(F.col("HourlyPrecipitation"), "s$", "").cast(DoubleType()))
```

CSV라 모든 컬럼이 문자열로 들어오는데(→ [CSV는 기본적으로 전부 String으로 읽힌다](#csv는-기본적으로-전부-string으로-읽힌다)), 그 문자열 안에 관측 데이터 특유의 표기가 섞여 있어서 바로 숫자로 못 바꾼다. 여기서 쓴 패턴은 Spark로 지저분한 실데이터를 정제할 때 자주 쓰는 조합이다.

- `rlike(정규식)`: 값이 특정 형식(정수 형태 `^-?\d+$`)에 맞는지 검사 — 형식에 안 맞으면(`"*"` 같은 결측 마커) `F.when`의 조건이 거짓이 되고, `otherwise`가 없으므로 자동으로 `null`이 됨
- `isin(...)`: 여러 개의 특수값(`"T"`=trace, `""`=빈 문자열)을 한 번에 매칭
- `regexp_replace(컬럼, "s$", "")`: 문자열 **끝**에 붙은 추정치 플래그(`s`)만 정규식으로 떼어내고 나머지는 그대로 둠
- 마지막에 `.cast(DoubleType())`으로 명시적 형변환

### date_trunc으로 두 데이터셋을 시간 단위로 조인

```python
hourly_trips_df = trips_df.withColumn("obs_hour", F.date_trunc("hour", "tpep_pickup_datetime")) ...
hourly_weather_df = ... .withColumn("obs_hour", F.date_trunc("hour", F.to_timestamp("DATE"))) ...
joined_df = hourly_trips_df.join(hourly_weather_df, on="obs_hour", how="inner")
```

`F.date_trunc("hour", 타임스탬프)`는 분/초를 버리고 "정시"로 내림한다(예: `14:37:22` → `14:00:00`). 운행 기록과 기상 관측은 원래 서로 다른 정밀도로 찍혀 있어서(운행 기록은 초 단위, 기상 관측은 매시 51분경) 그대로는 조인 키가 절대 안 맞는다 — 양쪽 다 같은 방식으로 "시" 단위로 내림해야 비로소 같은 값(`obs_hour`)을 공유하는 조인 키가 생긴다. `how="inner"`를 쓴 이유는, 기상 데이터는 연도 전체인데 운행 데이터는 그보다 짧은 기간만 있어서, inner join 한 번으로 자연스럽게 "운행 데이터가 있는 기간"으로 좁혀지기 때문이다(양쪽에 다 있는 시각만 살아남음).

### Spark → pandas 경계에서 통계 검정 수행

```python
pdf = joined_df.select("trip_count", "temperature_f", "precipitation_in").na.drop().toPandas()
temp_corr, temp_pvalue = stats.pearsonr(pdf["temperature_f"], pdf["trip_count"])
ttest_stat, ttest_pvalue = stats.ttest_ind(rainy, dry, equal_var=False)
```

시간 단위로 이미 집계된(최대 수천 행) `joined_df`를 `toPandas()`로 드라이버 메모리에 가져와서 `scipy.stats`로 넘긴다. 이 경계선은 [3번 항목](#3-spark-집계-결과가-충분히-작아지면-driver로-모아-pandasscipy로-통계-검증)에서 이미 다뤘으니, 여기서는 **쓴 통계 방법 자체**를 설명한다.

- **`pearsonr(x, y)`**: x와 y 사이의 **피어슨 상관계수(r)**와 그 상관이 우연히 나올 확률(**p-value**)을 같이 반환한다. r은 -1(완전한 음의 선형관계)~+1(완전한 양의 선형관계) 사이 값이고, p-value가 작을수록(보통 0.05 미만) "이 상관관계가 우연이 아니다"라고 말할 근거가 강해진다. 단, 상관계수는 어디까지나 **선형** 관계만 잡아낸다 — 관계가 있어도 곡선형이면 r이 낮게 나올 수 있다.
- **`ttest_ind(a, b, equal_var=False)`**: 두 독립적인 그룹(a=비 오는 시간, b=안 오는 시간)의 **평균이 통계적으로 다른지** 검정한다. `equal_var=False`는 두 그룹의 분산이 같다고 가정하지 않는 **Welch's t-test**를 쓰겠다는 뜻 — 비 오는 시간(표본 수 적음)과 안 오는 시간(표본 수 훨씬 많음)의 분산이 실제로 다를 가능성이 높아서, 안전한 쪽(등분산 가정 안 함)을 선택했다.
- **상관계수와 t-검정은 답하는 질문 자체가 다르다**는 게 이 미션에서 실제로 드러난 핵심 포인트였는데, 그 구체적 사례는 [4번 항목](#4-상관계수와-t-검정은-다른-질문에-답한다-그리고-표본이-늘면-결론이-바뀔-수-있다)에 정리돼 있다.

# 새로 배운 핵심 개념

## 1. `spark.read.parquet(디렉터리)`는 확장자를 안 가리고 폴더 안 파일을 전부 parquet로 해석하려 한다

`clean_trips.py`는 처음에 `spark.read.parquet("/opt/data")`처럼 디렉터리를 통째로 읽었다. 나중에 같은 `data/` 폴더에 기상 데이터(`weather_2024_central_park.csv`)를 추가하자 다음 job 실행이 그대로 깨졌다.

```
[FAILED_READ_FILE.CANNOT_READ_FILE_FOOTER] ... weather_2024_central_park.csv ...
Caused by: file:/opt/data/weather_2024_central_park.csv is not a Parquet file.
```

`spark.read.parquet(경로)`는 그 경로 밑의 **모든 파일**을 parquet 포맷으로 가정하고 읽으려 하지, 확장자를 보고 걸러주지 않는다. 해결은 디렉터리 대신 글롭 패턴을 쓰는 것.

```python
# before
df = spark.read.parquet("/opt/data")
# after
df = spark.read.parquet("/opt/data/*.parquet")
```

**여러 파일 형식(운행 기록 parquet, 기상 CSV 등)을 같은 폴더에 두고 싶다면, 로딩 시점에 확장자 기반 글롭 패턴으로 명시적으로 좁혀야 한다.**

> **후속 변경**: 이 시점의 수정은 "TLC 데이터는 전부 Parquet"이라는 전제 위에서 날씨 CSV만 피해가는 정도였다. 나중에 미션 요구사항("CSV, Parquet 등 다양한 파일 형식 처리")을 다시 대조해보니, **TLC 운행 기록 자체가 CSV로 올 경우**는 여전히 못 읽는다는 걸 발견했다. 이걸 고친 과정은 [9번 항목](#9-요구사항을-스스로-재점검해서-찾은-진짜-격차--parquet-전용-로더와-합성-데이터로-검증하기)에 정리했다.

## 2. 이상치 필터링은 "음수만 거르기"로 부족하다 — 상한선도 같이 정해야 한다

미션 스펙엔 "음수 이동 시간/거리 필터링"이라고만 적혀 있었지만, 실제 2024-01 데이터를 훑어보니:

- `trip_distance` 자체엔 음수가 없었지만 **최댓값이 312,722마일**인 행이 있었다
- 파일 이름은 "2024-01"인데 `tpep_pickup_datetime` 최솟값이 **2002년**까지 내려갔다

이런 값을 그대로 두면 평균 같은 지표가 크게 왜곡된다. 그래서 하한(0 초과)뿐 아니라 상한도 같이 뒀다.

```python
MAX_DURATION_MINUTES = 24 * 60   # 뉴욕 시내 택시가 하루를 넘겨 운행하는 경우는 사실상 없다고 봄
MAX_DISTANCE_MILES = 100          # 뉴욕에서 인접 도시(필라델피아 등)까지도 100마일을 안 넘음

cleaned_df = df.filter(
    (F.col("trip_duration_minutes") > 0) & (F.col("trip_duration_minutes") <= MAX_DURATION_MINUTES) &
    (F.col("trip_distance") > 0) & (F.col("trip_distance") <= MAX_DISTANCE_MILES) & ...
)
```

**"의미 없는 값"은 음수뿐 아니라 "말도 안 되게 큰 값"의 형태로도 나타난다.** 실제 데이터를 미리 `pyarrow`/`pandas`로 min/max를 훑어보지 않았다면 이 이상치를 놓쳤을 것이다.

## 3. Spark 집계 결과가 충분히 작아지면 driver로 모아 pandas/scipy로 통계 검증

Spark `DataFrame.stat.corr()`는 피어슨 상관계수(r)만 주고 **p-value는 계산해주지 않는다.** 미션 요구사항("통계적 방법으로 검증")을 채우려면 유의확률이 필요해서, 시간 단위로 집계된 (최대 4,366행 정도의) 작은 결과만 `toPandas()`로 모아 `scipy.stats`를 썼다.

```python
pdf = joined_df.select("trip_count", "temperature_f", "precipitation_in").na.drop().toPandas()
temp_corr, temp_pvalue = stats.pearsonr(pdf["temperature_f"], pdf["trip_count"])
ttest_stat, ttest_pvalue = stats.ttest_ind(rainy, dry, equal_var=False)
```

**핵심은 `toPandas()`를 부르는 시점.** 수천만 행짜리 원본(`cleaned_trips`)에 바로 호출하면 driver 메모리가 터진다. 여기선 `groupBy("obs_hour")`로 이미 시간 단위까지 뭉친 뒤였기 때문에 안전했다. Spark는 "대용량 집계"까지만 담당하고, Spark에 없는 기능(유의성 검정)은 결과가 충분히 작아진 뒤 pandas/scipy 쪽 책임으로 넘기는 경계선을 그은 것.

## 4. 상관계수와 t-검정은 다른 질문에 답한다, 그리고 표본이 늘면 결론이 바뀔 수 있다

처음 1개월치(2024-01)만으로 검증했을 때:

| 지표 | 1개월(1월) | 6개월(1~6월) | 7개월(2023-12~2024-06) |
|---|---|---|---|
| 강수량 vs 운행건수 상관계수 | r=-0.055 (p=0.132, 유의하지 않음) | r=0.051 (p=0.0008, 유의함, **부호도 반전**) | r=0.021 (p=0.143, 유의하지 않음) |
| 비 오는 시간 vs 안 오는 시간 t-검정 | p=0.0035 (유의함) | p=0.567 (**유의하지 않음**) | p=0.886 (유의하지 않음) |

1개월치만 봤을 때는 "강수량 자체와의 선형 상관은 없지만, 비/눈이 오는지 여부로 나누면 운행건수가 유의하게 준다"는 그럴듯한 결론이 나왔다. 그런데 데이터를 6개월로 늘리자 이 t-검정 결과가 뒤집혔고, 12월 한 달치를 더 얹어 7개월로 늘려도 "유의하지 않다"는 결론이 그대로 유지됐다 — 표본이 늘어날수록 1개월치 결론(우연한 패턴)에서 점점 멀어지고 있다는 뜻으로 읽힌다.

**교훈 두 가지**
- 연속값의 선형 상관관계(`pearsonr`)와 그룹을 나눠서 평균을 비교하는 것(`ttest_ind`)은 답하는 질문 자체가 다르다. 하나가 유의하지 않다고 다른 하나도 유의하지 않은 게 아니다.
- 표본 기간이 짧으면(한 달) 우연한 패턴이 통계적으로 유의한 것처럼 보일 위험이 있다. 데이터 기간을 늘려 재검증하기 전까지는 "유의하다"는 결론도 잠정적으로 봐야 한다.

## 5. 실제 공개 기상 데이터(NOAA LCD)는 한 파일에 여러 리포트 타입/지저분한 값이 섞여 있다

NOAA Local Climatological Data CSV 하나에 `REPORT_TYPE`이 `FM-15`(시간별 관측), `FM-16`, `SOD`(일별 요약), `SOM`(월별 요약)까지 섞여 있어서, 시간별 관측만 골라 쓰려면 먼저 걸러야 했다.

```python
hourly_weather_df = weather_raw_df.filter(F.col("REPORT_TYPE").contains("FM-15"))
```

값 자체도 지저분했다.
- `HourlyPrecipitation`: `"T"`(trace, 극미량), `"0.05s"`(뒤에 추정치 플래그 `s`가 붙음), 빈 문자열
- `HourlyDryBulbTemperature`: 대부분 정수 문자열이지만 결측은 `"*"`로 표기

이걸 문서만 보고 짐작하지 않고, 실제로 로컬에서 `csv.DictReader`로 `REPORT_TYPE` distinct 값과 정규식에 안 걸리는 값들을 먼저 뽑아본 뒤에야 파싱 로직(`F.when`/`regexp_replace`)을 확정했다.

**"공식 문서에 적힌 컬럼 설명"과 "실제 파일에 들어있는 값의 형태"는 다를 수 있다. 파싱 로직을 짜기 전에 실제 데이터를 소량이라도 직접 눈으로 훑어보는 단계를 생략하면 안 된다.**

## 7. Jupyter를 마스터/워커와 같은 이미지로 재사용하되, entrypoint만 오버라이드

시각화용 Jupyter Lab을 별도 이미지(`jupyter/pyspark-notebook` 등) 대신 **마스터/워커와 같은 Dockerfile**로 만들었다. 이미지의 `ENTRYPOINT`(`entrypoint.sh`)는 `SPARK_MODE`가 `master`/`worker`가 아니면 에러 내고 종료하는 구조라, 그대로 두면 Jupyter 컨테이너가 못 뜬다. `docker-compose.yml`에서 `entrypoint`만 비우고 `command`로 `jupyter lab`을 직접 실행하도록 오버라이드했다.

```yaml
jupyter:
  build:
    context: .
    dockerfile: Dockerfile   # 마스터/워커와 동일 이미지
  entrypoint: []             # Dockerfile의 ENTRYPOINT(entrypoint.sh)를 무시
  command: >
    jupyter lab --ip=0.0.0.0 --port=8888 ...
```

같은 이미지를 쓰는 이유는 **Spark/파이썬 패키지 버전을 클러스터와 항상 동일하게 유지**하기 위해서다. 그래야 노트북 안에서 `SparkSession.builder.master("spark://spark-master:7077")`로 클러스터에 직접 붙어 인터랙티브하게 분석해도 버전 불일치 문제가 안 생긴다.

## 8. Standalone 클러스터는 별 설정 없이도 두 워커에 작업을 나눠 분산한다

워커 코어/메모리(`SPARK_WORKER_CORES` 등)를 따로 설정하지 않고, `spark-submit`에도 `--total-executor-cores` 같은 옵션 없이 기본값으로 제출했다. "정말 워커 2대를 다 쓰고 있는지" 실행 로그로 직접 확인해봤다.

```bash
grep -oE "\(172\.19\.0\.[0-9]+,executor [0-9]+" 로그 | sort | uniq -c
#   37 (172.19.0.3,executor 1
#   38 (172.19.0.5,executor 0
```

두 워커 컨테이너(`172.19.0.3`, `172.19.0.5`)에 태스크가 37:38로 고르게 분산돼 있었다. **Standalone 모드에서는 별다른 리소스 옵션을 안 줘도, 마스터가 사용 가능한 워커 전부에 executor를 하나씩 배정한다.** 워커 대수를 늘려도 코드는 그대로 두면 되고(파티션 수는 데이터 크기에 따라 자연스럽게 늘어남), "워커를 쓰고 있다"는 걸 확인하고 싶으면 로그의 executor ID/IP를 세어보면 된다.

<br>

## 9. 요구사항을 스스로 재점검해서 찾은 진짜 격차 — Parquet 전용 로더와 합성 데이터로 검증하기

기능 요구사항을 다 채웠다고 생각했는데, 나중에 원본 스펙 문구("CSV, Parquet 등 다양한 파일 형식을 처리할 수 있습니다")를 다시 대조해보니 실제로는 안 채워진 부분이 있었다. `clean_trips.py`는 날씨 CSV를 피하려고 [1번 항목](#1-sparkreadparquet디렉터리는-확장자를-안-가리고-폴더-안-파일을-전부-parquet로-해석하려-한다)에서 `/opt/data/*.parquet`로 고쳤을 뿐, **TLC 운행 기록 자체가 CSV로 오면 여전히 못 읽었다.**

고친 방식은 파일명 패턴으로 두 포맷을 각각 찾아 `unionByName`으로 합치는 것.

```python
parquet_paths = sorted(glob.glob(f"{input_dir}/yellow_tripdata_*.parquet"))
csv_paths = sorted(glob.glob(f"{input_dir}/yellow_tripdata_*.csv"))

dfs = []
if parquet_paths:
    dfs.append(spark.read.parquet(*parquet_paths))
if csv_paths:
    dfs.append(spark.read.option("header", True).option("inferSchema", True).csv(*csv_paths))

df = dfs[0]
for other in dfs[1:]:
    df = df.unionByName(other)
```

이 로직이 진짜 맞는지는 말로 설명하는 것보다 **숫자로 확인하는 게 훨씬 확실했다.** 기존 parquet 데이터에서 500행만 잘라 합성 CSV(`yellow_tripdata_9999-99.csv`)를 만들어 같은 폴더에 넣고 돌려봤더니, `raw_count`가 기존 6개월치(20,332,093) 대비 **정확히 500만큼** 늘어난 20,332,593이 나왔다 — 두 포맷이 실제로 합쳐지고 있다는 걸 정량적으로 확인한 것. 테스트가 끝난 뒤엔 합성 파일을 지우고 다시 돌려서 공식 결과(20,332,093)를 복구했다.

**교훈 두 가지**
- "기능 요구사항 문구를 코드가 다 만족하는가"는 구현 직후가 아니라, 한 번 더 스펙과 나란히 놓고 재점검해야 놓친 부분이 보인다. (여기선 "다양한 형식"이라는 말을 "날씨 데이터가 CSV니까 됐다"고 스스로 넘겨짚었었다.)
- 포맷 자동 감지 같은 로직은 실제 데이터를 섞어 넣고 **정확한 개수 변화량**으로 검증하는 게, "에러 없이 실행됐다"보다 훨씬 신뢰할 수 있는 증거다.

<br>

## 10. "에러 없이 돈다"와 "의도한 데이터를 다 반영했다"는 다르다 — 조인 결과 행 수 변화로 검증

연 단위 확장도 비슷한 문제가 있었다. 처음엔 `download_data.sh`의 월 계산 로직이 연도 경계(예: 2023-11→2024-02)를 올바르게 순회하는지만 로컬에서 드라이런으로 확인했는데, 이건 "스크립트가 맞는 달 목록을 만드는가"만 검증한 것이지 **"실제 여러 해 데이터로 파이프라인 전체가 맞게 동작하는가"는 검증한 게 아니었다.**

그래서 실제로 2023년 12월 TLC 데이터와 2023년 기상 데이터를 받고, `weather_correlation.py`의 기본 경로도 단일 파일(`weather_2024_central_park.csv`)에서 연도별 파일을 전부 읽는 글롭 패턴(`weather_*_central_park.csv`)으로 바꿔서 7개월(2023-12~2024-06, 2개 연도)로 4개 job을 전부 재실행했다.

여기서도 "에러 없이 끝났다"만으로는 부족했다. 만약 글롭이 잘못돼서 `weather_2024_central_park.csv` 하나만 읽혔어도 job은 에러 없이 성공했을 것이다(2023-12 시간대만 조인에서 자연스럽게 빠질 뿐이니까). 그래서 **조인 결과 행 수의 변화량**을 근거로 삼았다.

```
6개월(2024-01~06) 기준 joined hourly records: 4,366
7개월(2023-12~2024-06) 기준 joined hourly records: 5,112   (+746)
```

12월 한 달은 대략 31일 × 24시간 ≈ 744시간이라, 늘어난 746과 거의 정확히 일치한다. 이 숫자가 딱 맞아떨어진다는 것 자체가 `weather_2023_central_park.csv`가 실제로 읽혀서 12월 시간대와 제대로 조인됐다는 증거다.

**"코드가 에러 없이 실행된다"는 "의도한 데이터를 빠짐없이 반영했다"를 보장하지 않는다.** 특히 `join`처럼 한쪽에 매칭되는 데이터가 없어도 조용히(에러 없이) 결과가 줄어들기만 하는 연산은, 결과 건수의 기대값을 미리 계산해두고 실제 결과와 대조하는 습관이 필요하다.

## `CANNOT_READ_FILE_FOOTER`: parquet 아닌 파일을 parquet로 읽으려 함

- **증상**: `submit.sh`를 job 이름 기반으로 일반화한 뒤 회귀 테스트로 `clean_trips`를 다시 돌렸더니 `FAILED_READ_FILE.CANNOT_READ_FILE_FOOTER` 에러로 실패
- **원인**: [1번 항목](#1-sparkreadparquet디렉터리는-확장자를-안-가리고-폴더-안-파일을-전부-parquet로-해석하려-한다) 참고 — `spark.read.parquet("/opt/data")`가 같은 폴더의 `weather_2024_central_park.csv`까지 parquet로 읽으려 시도
- **해결**: `clean_trips.py`의 기본 입력 경로를 `/opt/data/*.parquet` 글롭 패턴으로 변경
- **되짚어볼 점**: `submit.sh` 일반화 자체는 사용자 인터페이스만 바꾸는 간단한 리팩터링이었지만, 리팩터링 후 4개 job을 실제로 다시 돌려보는 회귀 테스트 과정에서 이 버그가 드러났다. 코드를 안 건드렸다고 재테스트를 생략했다면 이 버그를 계속 몰랐을 것.
