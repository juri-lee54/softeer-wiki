## 1. 리뷰 (사실 정리 & 평가)

#### 오늘의 한 일
- W3M1 
- W3M2a 

#### 오늘의 핵심 성과
- W3M2a를 W3M1 기반으로 다시 짜면서, Hadoop 다운로드 미러를 `archive.apache.org` → `dlcdn.apache.org`로 바꿔 빌드 시간을 20분 이상에서 수십 초 수준으로 줄임

#### 오늘의 핵심 이슈
- W3M2a를 처음 빌드했을 때 Hadoop 다운로드가 20분 넘게 걸림 → 원인은 `Dockerfile`이 느린 아카이브 미러(`archive.apache.org`)를 사용하고 있었기 때문
- `docker-compose.yml`의 3개 서비스가 같은 이미지 태그(`hadoop-cluster:latest`)를 동시에 빌드하면서 `already exists` 에러로 일부 서비스 빌드가 실패 표시되는 경합(race) 현상을 마주침 (이미지 자체는 정상 생성되어 실질적 문제는 없었음)
- 호스트 zsh에서 `docker exec`으로 MapReduce 명령을 바로 실행했을 때, `$HADOOP_HOME`과 `*` 와일드카드를 zsh가 먼저 해석하려다 `no matches found` 에러 발생 → 명령 전체를 따옴표로 감싸 컨테이너 내부 bash가 해석하도록 수정해서 해결

## 2. 회고 (KPT — 과정 개선)

#### Keep (잘해서 계속 유지할 행동)
- 기존에 잘 동작하던 코드(W3M1)를 재사용해서 새 미션(W3M2a)의 리스크를 줄인 것 — 멀티노드 로직만 추가하니 디버깅 범위가 좁아짐

#### Problem (문제가 발생한 행동)
- W3M2a 빌드가 오래 걸리는 걸 바로 의심하지 않고 그냥 기다렸음 — Dockerfile의 다운로드 URL을 먼저 확인했으면 더 빨리 원인을 찾았을 것

#### Try (다음에 새롭게 시도할 행동)
- Docker 이미지 빌드에서 외부 다운로드가 포함되면, 빌드 전에 미러 URL의 응답 속도를 먼저 확인하는 습관 들이기
- 여러 서비스가 같은 이미지 태그를 공유하는 `docker-compose.yml`을 짤 때는 병렬 빌드 경합 가능성을 미리 인지하고, 필요하면 순차 빌드나 `docker build` 단독 실행으로 우회하기
