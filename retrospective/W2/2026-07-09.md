#### 오늘의 핵심 이슈

- 인스턴스 유형을 t2.small로 바꾸는 과정에서 **퍼블릭 IP가 바뀌고, https/http 혼동으로 접속이 안 되는** 부수적인 문제가 또 발생함 → 매번 인스턴스 상태가 바뀔 때마다 "IP가 그대로인가, Elastic IP가 잘 붙어있는가"를 확인하는 습관이 아직 부족함
- 팀 과제 피드백에서 **"과정은 좋았는데 결론이 미약하다"**는 지적을 받음. 사전에 정의한 비즈니스 가치(부정 리뷰 원인 파악 → 우선순위 패치 계획)까지 결론을 일치시키지 못하고, 미션을 푸는 것 자체에서 멈춤. 또한 모든 댓글을 동일한 가치로 평가하면 안 된다는 것(주제 관련성 판별 필요)도 놓친 부분으로 확인됨

## 2. 회고 (KPT)

#### Keep

- 어제 "Try"에 적어둔 것(메모리 확정 검증 → swap 또는 스펙 상향 선택)을 오늘 바로 실행해서 실제로 문제를 해결한 것. 회고가 그냥 기록으로 끝나지 않고 다음 날 행동으로 이어짐
- Dockerfile을 처음부터 완벽하게 짜려고 하지 않고, 문제를 하나씩 만나면서 점진적으로 고쳐나간 접근 방식 (오늘 배운 "Jupyter Notebook으로 먼저 한 줄씩 검증 후 Dockerfile화"하는 방식과도 맥이 닿아 있음)

#### Problem

- 인스턴스 스펙/상태가 바뀔 때마다(t2.micro→t2.small 등) 그로 인해 파생되는 부수적 문제(퍼블릭 IP 변경, 프로토콜 혼동)를 매번 그때그때 대응하고 있음 — 체크리스트 없이 진행하다 보니 같은 유형의 실수(주소/프로토콜 확인 누락)가 반복됨
- 팀 과제에서 "기술적으로 요구사항을 충족시키는 것"과 "비즈니스 의사결정자가 봤을 때 설득력 있는 결론을 내는 것"을 분리해서 생각하지 못했음. 사전 비즈니스 가치 정의는 잘했는데, 결과물이 그 질문에 답하는 형태로 마무리되지 않음

#### Try

- 인스턴스 재구성(유형 변경, 재부팅 등) 후에는 **"IP 확인 → 프로토콜(http) 확인 → 컨테이너 실행 상태 확인"**을 순서대로 체크하는 나만의 체크리스트를 만들어서 다음부터 바로 확인하기
- 데이터 파일을 Dockerfile에 포함시켜 재빌드 → ECR 재push → EC2 재pull까지 마무리해서, 노트북이 실제로 끝까지 정상 실행되는지 최종 확인하기
- 다음 팀 과제부터는 **"이 결론이 사전에 정의한 비즈니스 가치 질문에 답하고 있는가"**를 마지막에 스스로 점검하는 단계를 넣기. 예: "부정 리뷰의 원인을 파악한다"고 했으면, 결론도 그 원인과 패치 우선순위로 끝나야 함
- 리뷰/댓글 분석처럼 정성적 데이터를 다룰 때는, 모든 데이터를 동일 가중치로 다루지 않고 **주제 관련성 판별 단계**를 먼저 거치는 습관 들이기
