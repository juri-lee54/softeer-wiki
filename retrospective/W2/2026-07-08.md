# 일일 회고 - 2026-07-08

## 1. 리뷰 (사실 정리 & 평가)

#### 오늘의 한 일

- AWS EC2 인스턴스 생성 (Ubuntu, t2.micro, 프리티어)
- 보안 그룹에 22번(SSH), 8888번(JupyterLab) 포트 개방
- User-data 스크립트로 Docker 자동 설치 시도 → IPv6 네트워크 문제로 실패 → `ForceIPv4` 설정 추가해서 해결
- Elastic IP를 할당해서 EC2에 고정 퍼블릭 IP 연결
- IAM 역할/Access Key 생성 시도 → 교육용 계정 권한 제한으로 계속 실패
- MFA 기반 임시 자격증명(`sts get-session-token`)으로 우회 인증 시도
- 로컬 Docker 이미지를 ECR에 push (성공)
- EC2에서 ECR pull 시도 → SCP(조직 정책)에 의해 차단 → 관리자에게 EC2용 IAM 역할 부여 요청해서 해결
- 이미지 아키텍처 불일치(arm64 vs amd64) 발견 → `--platform linux/amd64`로 재빌드하여 해결
- EC2 디스크 용량 부족(`no space left on device`) → EBS 볼륨 20GB로 확장 후 해결
- 컨테이너 실행 및 브라우저로 JupyterLab 접속 확인
- 노트북에 데이터 파일이 빠져있는 것을 발견 (해결 방법은 확인했으나 아직 재배포는 안 함)
- 노트북 실행 중 커널이 죽는(Kernel Restarting) 문제 발생 → 메모리 부족(OOM) 추정, **원인 조사만 하고 해결하지 못한 채 하루 마감**

#### 오늘의 핵심 성과

- EC2 ↔ Docker ↔ ECR로 이어지는 전체 배포 파이프라인을 처음부터 끝까지(권한 문제 포함) 완주해서, 브라우저로 JupyterLab에 접속하는 것까지 확인함
- 하루 동안 겪은 문제들이 전부 성격이 다른 레이어(네트워크, 권한/조직정책, 아키텍처, 스토리지)에서 나왔는데, 각각을 원인 분리해서 하나씩 격파함
- 특히 SCP처럼 개인 권한 문제인 줄 알았다가 조직 정책 문제라는 걸 알아낸 것 — 겉으로 보이는 에러 메시지만 보고 넘어가지 않고 근본 원인까지 추적한 게 좋았음

#### 오늘의 핵심 이슈

- 커널 재시작(Kernel Restarting) 문제를 해결하지 못하고 하루가 끝남. t2.micro의 메모리(1GB)가 부족해서 OOM으로 커널이 죽는 것으로 추정했지만, `free -h`나 `dmesg` 로그로 확정 원인을 검증하지 못했고, swap 추가 등의 해결책도 실제로 적용해보지 못한 채 미해결 상태로 남음
- 데이터 파일 누락 문제도 Dockerfile 수정 방향만 잡고 실제 재빌드/재배포는 다음으로 미룸

## 2. 회고 (KPT)

#### Keep

- 에러 메시지를 그냥 넘기지 않고 매번 "왜 이 에러가 나는지"를 원인 레벨까지 파고든 것 (IPv6 라우팅, SCP, 아키텍처 불일치 등 증상 뒤의 원인을 찾아냄)
- 문제를 겪을 때마다 바로바로 정리 문서에 기록해서, 나중에 무엇을 언제 어떻게 해결했는지 추적 가능하게 해둔 것

#### Problem

- 문제를 하나씩 순차적으로 해결하다 보니, 뒤에 있는 문제(커널 재시작)까지 도달했을 때 시간/에너지가 부족해서 충분히 검증하지 못하고 "추정"만 하고 끝남
- 인스턴스 스펙(t2.micro의 메모리 한계)을 사전에 고려하지 않고 진행하다가, 문제가 실제로 터진 뒤에야 스펙 문제를 인지함

#### Try

- 내일은 `free -h`, `dmesg | grep -i "killed process"`로 먼저 메모리 부족이 확정 원인인지 검증하고 시작
- swap 추가 또는 인스턴스 유형 상향(t2.micro → t2.small) 중 하나를 선택해서 커널 재시작 문제를 실제로 해결
- 데이터 파일을 Dockerfile에 포함시켜 재빌드 → ECR 재push → EC2 재pull까지 마무리해서, 노트북이 실제로 끝까지 정상 실행되는지 확인
- 다음번엔 인스턴스를 만들기 전에 예상 워크로드에 맞는 스펙(메모리 등)을 먼저 가늠해보고 시작하기
